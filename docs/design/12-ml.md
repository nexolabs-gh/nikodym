## 1. Propósito y responsabilidad

**Qué resuelve (una frase).** `ml` entrena un **modelo challenger de machine learning** (SVM, Random Forest, XGBoost, LightGBM o CatBoost, seleccionable por config) que **reta** al scorecard logístico campeón (SDD-08/09) sobre el **mismo pipeline de datos**, produce su PD y una **comparación cabeza-a-cabeza** challenger-vs-campeón, sin reemplazar al modelo primario regulatorio.

**Responsabilidad única (qué SÍ hace).**
- Expone un **wrapper de estimador genérico** (`MLChallenger`) con una interfaz común sklearn-like (`fit`/`predict_proba`/`predict_pd`/`get_params`/`set_params`) sobre **backends intercambiables por config**. La abstracción del backend es **target-agnóstica por diseño** (reusable a futuro para LGD/EAD), pero v1 solo cablea **clasificación binaria / PD**.
- Consume las features WoE ya calculadas por `binning` (SDD-06) —o, condicionalmente, las seleccionadas por `selection` (SDD-07)— para competir manzana-con-manzana con el campeón; **no re-inventa el preprocesamiento**.
- Ajusta el challenger en la partición de desarrollo, predice PD en todas las particiones y **alinea por índice** con la PD cruda del campeón (`("model","raw_pd_frame")`).
- Calcula la **comparación de discriminación y estabilidad** challenger-vs-campeón **reusando** `PerformanceEvaluator`/`StabilityEvaluator` de SDD-11 (AUC/Gini/KS/PSI/CSI); nunca reimplementa una métrica.
- Aplica **restricciones de monotonía** (`monotonic constraints`) donde el backend lo soporta (intrínseco al wrapper, principio 4 de ESPECIFICACIONES).
- Opcionalmente **calibra** la PD del challenger reusando `PDCalibrator` (SDD-10) y la publica en el formato de `calibrated_pd_frame`, para que `validation` (SDD-22) y `report` (SDD-26) la consuman como consumen la del campeón.
- Publica artefactos namespaced bajo `"ml"`: estimador fiteado, PD del challenger, comparación, metadata del backend, resultado y card CT-2.
- Aporta `MLConfig` como sección **computacional** de `NikodymConfig`; por tanto, cambios de backend, hiperparámetros o fuente de features mueven el `config_hash`.
- Registra con `log_decision` el backend elegido, la semilla derivada, las constraints de monotonía aplicadas, la comparación con el campeón y cada gate de reproducibilidad/determinismo.

**Límites explícitos (qué NO hace, y quién lo hace).**
- **No optimiza hiperparámetros ni corre Optuna.** El tuning (samplers seedeados, search spaces editables) es SDD-13 `tuning`, que consume la API sklearn-like de este wrapper (ESPECIFICACIONES.md:L191, ROADMAP.md:L69). `ml` recibe hiperparámetros ya fijados por config.
- **No calcula SHAP ni reason codes.** La explicabilidad (SHAP + reason codes, que sustituyen a la scorecard en modelos no lineales) es SDD-14 `explain`, transversal a scorecard y ML (ESPECIFICACIONES.md:L119, L192). `ml` solo expone **feature importances nativas** del backend (gain/split), no post-hoc.
- **No reimplementa métricas de discriminación/estabilidad.** AUC/Gini/KS/PSI/CSI salen de SDD-11 (`PerformanceEvaluator`/`StabilityEvaluator`); `ml` los llama, no los recodifica.
- **No ejecuta tests formales de calibración.** Hosmer-Lemeshow, binomial/Jeffreys por grado, Brier y traffic-light pertenecen a SDD-22 (`validation`), no a SDD-10/11 ni a `ml` (11-performance-stability.md:L43, 10-calibration.md:L34).
- **No reemplaza al scorecard como modelo primario.** El scorecard logístico es interpretable/regulatorio y sigue siendo el campeón; ML aporta **benchmark de poder predictivo** (ESPECIFICACIONES.md:L121). No son intercambiables 1:1.
- **No modela LGD/EAD en v1.** Esos parámetros usan econometría propia (beta regression / fractional / Tobit / workout) en `provisioning/ifrs9` de F4 (ESPECIFICACIONES.md:L135-L136), no ML. La abstracción de `ml` se diseña extensible a ese objetivo, pero solo PD/binario se implementa.
- **No hace reject inference, feature engineering ni selección de variables.** Consume features aguas arriba; el MVP es scorecard de comportamiento (AGENTS.md, decisiones fijadas).
- **No trae la capa de datos longitudinal (CT-3).** Opera sobre el **panel transversal de scorecard** (SDD-02); IFRS 9/forward traen su propio panel.
- **No usa `eval` ni expresiones ejecutables de usuario.** Los hiperparámetros son estructuras Pydantic tipadas por backend.
- **No promete determinismo byte-a-byte multihilo.** Los GBDT paralelos no son 100% deterministas (ESPECIFICACIONES.md:L77); v1 default a single-thread determinista y marca el modo performance como no byte-reproducible.

## 2. Contexto y ubicación en la arquitectura

- **Capa:** Machine Learning (F2/T3), inmediatamente después del pipeline de scorecard F1 (datos → binning → selección → modelo → scorecard → calibración → performance/stability). Reusa ese pipeline, no lo duplica (ROADMAP.md:L71 DoD "mismo pipeline de datos que F1").
- **Quién lo invoca:** `Study.run()` como sección `ml` de `NikodymConfig`, o API programática `MLChallenger.from_config(...)` + `MLStep`.
- **A quién invoca:** `core` (`Step`, `ArtifactKey`, `ArtifactStore`, `NikodymClassifier`, `AuditableMixin`, `SeedManager`, excepciones, lineage); `binning`/`selection` (features WoE); `model` (PD del campeón); y —reusados programáticamente— `PerformanceEvaluator`/`StabilityEvaluator` (SDD-11) y `PDCalibrator` (SDD-10).
- **Quién lo consume aguas abajo:** SDD-13 (`tuning`) envuelve el wrapper para buscar hiperparámetros; SDD-14 (`explain`) explica el estimador fiteado con SHAP; SDD-22 (`validation`) valida la PD calibrada del challenger; SDD-26 (`report`) renderiza la comparación campeón-vs-ML (ROADMAP.md:L70).

```text
data.labels + data.splits ─┐
binning.woe_frame ─────────┼──► ml  ──► ml.pd_frame (PD challenger)
model.raw_pd_frame ────────┘     │     ├─► ml.comparison (challenger vs campeón)
  (PD campeón, para comparar)    │     ├─► ml.calibrated_pd_frame (opcional → validation/report)

     reúsa PDCalibrator (opcional)
```

**Anclaje normativo/metodológico interno.**
- ESPECIFICACIONES §5.3 fija F2 como benchmark ML: SVM/RF/XGBoost/LightGBM/CatBoost (énfasis CatBoost), monotonic constraints donde aplique, y "ML aporta benchmark de poder predictivo" frente al scorecard interpretable/regulatorio (ESPECIFICACIONES.md:L116-L121).
- ESPECIFICACIONES §6.3 ubica el paquete `ml/` en el árbol y separa `tuning/` y `explain/` como paquetes hermanos (ESPECIFICACIONES.md:L190-L192).
- ROADMAP F2 declara el objetivo (benchmark + explicabilidad), los SDD (12 ml · 13 tuning · 14 explain), los entregables (wrappers con monotonic constraints, Optuna en tuning, SHAP + comparativa scorecard-vs-ML en el reporte) y el DoD (mismo pipeline F1; tuning reproducible; SHAP integrado; tests de determinismo) (ROADMAP.md:L64-L72).
- Principio 4 (monotonía con el riesgo) aplica a modelos ML vía monotonic constraints (ESPECIFICACIONES.md:L80); principio 9 (núcleo liviano) exige backends ML tras extras opcionales con import perezoso (ESPECIFICACIONES.md:L85, L235).
- CT-1 exige `requires`/`provides`; CT-2 exige card estructurada aditiva (`metric_sections`); CT-3 mantiene el panel transversal.

**Interacción con `Study` y config declarativo.** `MLStep` es un `Step` nativo registrado con `@register("standard", domain="ml")`. Se construye desde `MLConfig`; sus `requires` son **dinámicos** según `feature_source`:
- siempre requiere `("data","labels")` y `("data","splits")`;
- siempre requiere `("model","raw_pd_frame")` (PD del campeón para la comparación);
- si `feature_source="binning_woe"` (default): requiere `("binning","woe_frame")`;
- si `feature_source="selection_woe"`: requiere `("selection","selected_woe_frame")` y `("selection","selected_woe_columns")`;
- si `feature_source="data_raw"` (diferido/experimental): requiere `("data","frame")`.

Esto cumple CT-1: el objeto `MLStep` expone el DAG efectivo antes de ejecutar. El motor v1 valida prerequisitos; el scheduler topológico real sigue diferido a F5 por CT-1. Con la fuente default (`binning_woe`), las dependencias duras de build son SDD-02/06/08 (+ transversales 01/05), coherente con la columna "Depende de" del índice (06, 08); `selection` (07) es dependencia **condicional** solo bajo el modo apples-to-apples.

- `_DOMAIN_MODULES["ml"] = "nikodym.ml"`;

## 3. Conceptos y fundamentos

**Fuente normativa/metodológica interna.**
- ESPECIFICACIONES §5.3 (algoritmos, monotonía, SHAP delegado, benchmark) (ESPECIFICACIONES.md:L116-L121).
- ESPECIFICACIONES §3/§4 reproducibilidad: `random_state` en sklearn/XGBoost/LightGBM/CatBoost y caveat GBDT multihilo no 100% determinista (ESPECIFICACIONES.md:L77).
- ROADMAP F2 objetivo/entregables/DoD (ROADMAP.md:L64-L72).
- CT-1/CT-2/CT-3 (`_CONTRATOS-TRANSVERSALES.md`).

**Campeón vs challenger.** El **campeón** es la logística PD de SDD-08 (interpretable, regulatoria); su PD cruda vive en `("model","raw_pd_frame")` (columnas `partition`, `target`, `linear_predictor`, `pd_raw`; **no calibrada**, D-MOD-8 de SDD-08). El **challenger** es un modelo ML de mayor capacidad. La pregunta que responde `ml` es: **¿cuánto poder predictivo deja el scorecard sobre la mesa?** Se mide comparando discriminación (AUC/Gini/KS) y estabilidad (PSI) de ambos sobre las **mismas particiones**, con las **mismas etiquetas**.

**Discriminación es invariante a la calibración.** AUC/Gini/KS son métricas de **ordenamiento** (ranking); una transformación monótona de la PD (como la calibración por offset) no las altera. Por eso la comparación cabeza-a-cabeza usa **PD cruda de ambos** (`pd_raw` del campeón vs `pd_hat` del challenger) y es metodológicamente correcta sin calibrar. La calibración solo se necesita para comparar **nivel** de PD (tasa central) o para alimentar validación/provisiones, y por eso es un paso **opcional** (`calibrate_challenger`).

**Backends y su interfaz común.** Cada backend implementa el mismo protocolo `Backend` que `MLChallenger` orquesta:

| Backend | Librería | Extra | Monotonía | Notas |
|---|---|---|---|---|
| `svm` | scikit-learn (`SVC(probability=True)`) | `[ml]` | ✗ no soportada | probabilidades vía Platt interno; caro en N grande |
| `random_forest` | scikit-learn (`RandomForestClassifier`) | `[ml]` | ✗ no soportada | determinista con `random_state` |
| `xgboost` | xgboost | `[xgboost]` | ✓ `monotone_constraints` | GBDT; early stopping |
| `lightgbm` | lightgbm | `[lightgbm]` | ✓ `monotone_constraints` | GBDT; `deterministic=True` |
| `catboost` | catboost | `[catboost]` | ✓ `monotone_constraints` | GBDT; énfasis ESPEC (infrautilizado) |

Los backends que no soportan monotonía (`svm`, `random_forest`) **ignoran** las constraints con warning explícito (o fallan si `monotonic.on_unsupported="error"`); no se inventa una restricción que la librería no ofrece.

**WoE como representación de entrada (default).** El challenger consume por default las columnas WoE de `binning` (`<feature>__woe`). Ventajas: (i) **mismo preprocesamiento** que el campeón (missing/special/outliers ya resueltos por binning), (ii) comparación limpia (misma información de entrada), (iii) evita que `ml` reimplemente el manejo de valores especiales. Caveat metodológico: los árboles son **invariantes a transformaciones monótonas por variable**, de modo que el WoE aporta sobre todo **consistencia de pipeline**, no información adicional frente al bin ordinal; feeds de features **crudas** (`data_raw`) son un modo diferido que exige política de imputación propia (FALTA-DATO-ML-1). La monotonía "con el riesgo" se expresa sobre WoE recordando la convención de SDD-06 (§3): con `WoE = ln(%Goods/%Bads)` y `target=1=malo`, una event rate ascendente implica **WoE descendente** (un bin con más malos tiene WoE más bajo). Por tanto **WoE creciente ⇒ riesgo (PD) decreciente**, y la constraint por variable es **-1** en el espacio WoE (PD no creciente en WoE; ver §7).

**Partición interna de validación para early stopping.** El campeón fitea solo en `desarrollo`. Los GBDT usan early stopping, que requiere un conjunto de evaluación. Para **no contaminar** `holdout`/`oot` (que quedan como particiones de evaluación limpias), `ml` **recorta una fracción interna** de `desarrollo` (seeded, determinista) como set de early stopping. `holdout` y `oot` nunca se usan para entrenar ni para detener el entrenamiento.

**Notación.**
- `X_p`: matriz de features (WoE) de la partición `p ∈ {desarrollo, holdout, oot}`.
- `y_p`: vector de target binario (`1`=malo/default, `0`=bueno) de `p`.
- `X_fit`, `X_es`: sub-particiones de desarrollo para ajuste y early stopping.
- `f_θ`: challenger con hiperparámetros `θ` y semilla `s`.
- `pd_hat_{i}`: PD predicha del challenger para la observación `i` (índice pandas).
- `pd_raw_{i}`: PD cruda del campeón para `i`.
- `Δ_metric(p) = metric_challenger(p) − metric_campeón(p)`: brecha de una métrica en la partición `p`.

**Frameworks conceptuales.** No se embeben umbrales regulatorios de "cuánto mejor debe ser el challenger" (esos son FALTA-DATO-ML-2, del usuario/config). `ml` reporta las brechas `Δ_metric`; la decisión de gobierno (qué brecha justifica adoptar/monitorear el challenger) es del comité de modelos, no de la librería.

## 5. Configuración (schema Pydantic)

`MLConfig` es el sub-config de la sección `ml` de `NikodymConfig`. Sigue SDD-05: `NikodymBaseConfig`, `extra="forbid"`, `frozen=True`, `Literal` para categóricos, rangos explícitos, defaults defendibles y metadata UI. **`ml ∉ INFRA_SECTIONS`**: todo cambio computacional mueve `config_hash`.

**Impacto en `GOLDEN_DEFAULT_CONFIG_HASH` (verificado contra `core/config/hashing.py`).** `config_hash()` serializa `cfg.model_dump(mode="json", by_alias=True, exclude=INFRA_SECTIONS)` **sin** `exclude_none` (`core/config/hashing.py:L40`), y `INFRA_SECTIONS = {"name","governance","audit","tracking","report"}` (`:L24`) **no incluye `ml`**. Las secciones computacionales en `None` **sí entran** al payload canónico (confirmado por el golden vigente, que "incluye `stress=None` con cálculo independiente"). Por tanto, **añadir el campo `ml` (default `None`) a `NikodymConfig` agrega la clave `"ml":null` al JSON canónico y MUEVE `GOLDEN_DEFAULT_CONFIG_HASH`**. Esto es **esperado** al codificar B12.1: se recalcula y actualiza el golden en los tests de config, igual que hicieron `stress`/`validation`/`survival`. En este ítem solo-docs **no** se mueve ningún hash ni se toca código.

```python
# nikodym/ml/config.py
MLBackendName = Literal["svm", "random_forest", "xgboost", "lightgbm", "catboost"]
FeatureSource = Literal["binning_woe", "selection_woe", "data_raw"]

    n_estimators: int = Field(default=500, ge=1, le=10000)
    num_leaves: int = Field(default=31, ge=2, le=1024)
    learning_rate: float = Field(default=0.05, gt=0.0, le=1.0)
    subsample: float = Field(default=0.8, gt=0.0, le=1.0)
    colsample_bytree: float = Field(default=0.8, gt=0.0, le=1.0)
    reg_lambda: float = Field(default=1.0, ge=0.0)
    min_child_samples: int = Field(default=50, ge=1)

class CatBoostParams(NikodymBaseConfig):
    iterations: int = Field(default=500, ge=1, le=10000)

    fit_partition: str = "desarrollo"
    predict_partitions: tuple[str, ...] = ("desarrollo", "holdout", "oot")
```

**Resolución de hiperparámetros por backend.** Un `model_validator(mode="after")` coacciona `hyperparameters` al tipo correcto según `backend` (patrón análogo a la unión anidada por factory de `data`): si es `None`, instancia el modelo de defaults del backend; si es un `dict`, lo valida contra el modelo del backend; si es de otro backend, `MLConfigError`. Así el config no admite hiperparámetros de un backend distinto al seleccionado.

**Validaciones de config.**
- `hyperparameters` debe corresponder al `backend` seleccionado; incompatibilidad → `MLConfigError`.
- `deterministic=True` con `n_threads>1` → `MLConfigError` (el determinismo byte-a-byte de GBDT exige single-thread; ver §9). Alternativa: forzar `n_threads=1` con warning, según D-ML-4.
- `monotonic.mode="explicit"` con `explicit={}` → `MLConfigError`; claves deben ser features conocidas y valores en `{-1,0,1}`.
- `monotonic.mode != "off"` con backend sin soporte (`svm`/`random_forest`) → si `on_unsupported="error"`, `MLMonotonicError`; si `"warn"`, se ignora la constraint con `log_decision`.
- `feature_source="selection_woe"` exige que el DAG tenga `selection` aguas arriba (validado por CT-1 pre-run).
- `feature_source="data_raw"` es experimental: exige `imputation` declarada para backends que no toleran `NaN` (SVM/RF) → sin política, `MLConfigError` con `FALTA-DATO-ML-1`.
- `train.predict_partitions` y `comparison.partitions` deben ser subconjuntos de las particiones conocidas.
- `train.validation_fraction=0.0` deshabilita early stopping; con GBDT y `early_stopping_rounds` no nulo, `MLConfigError` (contradicción).

**Campos, defaults y sentido.**
- `backend`: default D-ML-1 `"xgboost"` (GBDT challenger canónico); requiere `nikodym[xgboost]`. Alternativa "corre con el extra base `[ml]`": `random_forest`. **default a confirmar por Cami.**
- `feature_source`: default D-ML-2 `"binning_woe"` (consistencia de pipeline y deps 06/08 exactas). `"selection_woe"` = apples-to-apples con el campeón; `"data_raw"` = diferido.
- `hyperparameters`: defaults D-ML-9 por backend, conservadores (árboles poco profundos, `learning_rate` bajo, regularización > 0) para un challenger honesto sin overfit; **a confirmar por Cami**, y de todos modos objeto del tuning de SDD-13.
- `monotonic.mode`: default D-ML-3 `"from_binning"` (deriva la dirección desde la tendencia monótona del binning; WoE creciente ⇒ constraint `-1`, PD no creciente en WoE).
- `train.validation_fraction`: default D-ML-7 `0.2`, recorte interno de desarrollo, seeded; `holdout`/`oot` intactos.
- `calibrate_challenger`: default D-ML-5 `False` (la comparación de discriminación no lo necesita; calibrar es opcional y solo para nivel/downstream).
- `train.class_weight`: default D-ML-6 `"none"` (no reponderar por default; `"balanced"` disponible).
- `deterministic`/`n_threads`: default D-ML-4 `True`/`1` (byte-reproducibilidad; ver §9).

**Round-trip YAML y UI.** El round-trip sigue SDD-05: dump JSON-mode, `sort_keys=False`, carga vía `load_config`. La UI (SDD-23) debe renderizar `backend` como selector, los hiperparámetros como formulario dependiente del backend, la monotonía como tabla feature→dirección, y advertir cuando `deterministic=False` (resultado no byte-reproducible).

**Hook diferido en `core.config.schema`.**
- declarar `_ML_CONFIG_CLS: type[BaseModel] | None = None`;
- añadir `ml` como campo `Any` en runtime y `MLConfig | None` bajo `TYPE_CHECKING`;
- añadir `@field_validator("ml", mode="before")` (`_valida_ml`) que coacciona vía el hook, con fallback a blob JSON-canónico si el hook no está cargado (patrón de `stress`/`survival`);
- no importar `nikodym.ml` desde `core`;
- al importar `nikodym.ml`, poblar `_schema._ML_CONFIG_CLS = MLConfig`;
- **actualizar `GOLDEN_DEFAULT_CONFIG_HASH`** en B12.1 (el campo `ml` queda dentro del hash canónico).

## 6. Contratos de datos (I/O)

**Input duro del Step según config.**

| artefacto | requerido | contrato |
|---|---|---|
| `("data","labels")` | siempre | `LabeledFrame` de SDD-02; provee `target_col` |
| `("data","splits")` | siempre | `PartitionResult` de SDD-02; provee `partition_col` |
| `("model","raw_pd_frame")` | siempre | DataFrame campeón: `partition`, `target`, `linear_predictor`, `pd_raw` (índice = filas modelables) |
| `("binning","woe_frame")` | si `feature_source="binning_woe"` | DataFrame WoE: índice, `partition`, `target`, columnas `<feature>__woe` |
| `("selection","selected_woe_frame")` + `("selection","selected_woe_columns")` | si `feature_source="selection_woe"` | subconjunto WoE seleccionado + nombres |
| `("data","frame")` | si `feature_source="data_raw"` | features crudas intactas (no-mutación SDD-02); requiere política de imputación |

Cada clave `requires` **existe como `provides`** en su SDD vecino: `data.labels`/`data.splits`/`data.frame` (SDD-02), `binning.woe_frame` (SDD-06 §prov), `selection.selected_woe_frame`/`selected_woe_columns` (SDD-07), `model.raw_pd_frame` (SDD-08 §prov). DAG cerrado.

**Alineación por índice (no hay `row_id`).** Ni `model` ni `binning` publican un `row_id` explícito: las filas modelables se identifican por el **índice pandas** compartido. `ml` fitea/predice sobre ese índice y hace el join campeón↔challenger por índice; si los índices no coinciden (distinto conjunto modelable), `MLComparisonError`. `ml` **no muta** los frames de entrada (copias defensivas).

**Poblaciones (heredadas del pipeline de scorecard).** Fit solo en `desarrollo` con `target` no nulo; predicción en `predict_partitions`; `fuera_de_modelo` se excluye. El early stopping usa un recorte seeded de `desarrollo` (no toca `holdout`/`oot`).

**Output `ml.pd_frame`.** DataFrame tidy, índice = filas modelables:

| columna | significado |
|---|---|
| `partition` | `desarrollo`/`holdout`/`oot` |
| `target` | etiqueta binaria (0/1), `NaN` donde no exista |
| `pd_hat` | PD del challenger (clase 1), en `[0,1]` |

**Output `ml.calibrated_pd_frame`** (si `calibrate_challenger=True`). Se ciñe a las **6 columnas core** del `calibration.calibrated_pd_frame` de SDD-10 (resumen de artefacto SDD-10 :252): `partition`, `target`, `linear_predictor` (logit de `pd_hat`), `pd_raw` (= `pd_hat` pre-calibración), `linear_predictor_calibrated`, `pd_calibrated`. El **método** y el **ancla** de calibración **no** son columnas del frame: viven en los `CalibrationParameters` que devuelve el `PDCalibrator` reusado (auditados y volcados al card), tal como SDD-10 los ubica. Ceñirse a esas 6 columnas hace inequívoco el "mismo esquema" y permite que `validation` (SDD-22) valide el challenger por su ruta genérica (ver nota de DAG abajo).

**Output `ml.comparison`.** `tuple[MLComparisonRecord, ...]`; su tidy (`comparison_frame()`): `partition`, `metric` (`auc`/`gini`/`ks`/`psi`), `champion_value`, `challenger_value`, `delta`, `better`, `source`.

**Invariantes.**
- `pd_hat ∈ [0,1]`; sin `NaN`/`inf`/`-inf`; `-0.0 → 0.0`.
- `pd_frame` conserva el índice y el orden de las filas modelables de la fuente de features.
- La comparación evalúa campeón y challenger sobre **las mismas filas y particiones** (mismo `PerformanceEvaluator`, misma config de columnas), de modo que `delta` sea apples-to-apples.
- `better` se decide con `comparison.tie_tolerance` (|delta| ≤ tol ⇒ `"tie"`); para `ks`/`psi` la orientación de "mejor" se documenta (mayor KS = mejor discriminación; menor PSI = más estable).
- No se mutan `data.*`, `binning.woe_frame`, `selection.*` ni `model.raw_pd_frame`.
- Orden estable: particiones en orden de config, métricas en orden de config, importances descendentes por valor con desempate lexicográfico por nombre.

**Nota de DAG aguas abajo (validación del challenger).** SDD-22 `validation` **no** declara `requires` sobre el dominio `ml`: valida un modelo a la vez y entra por la ruta genérica que consume `("calibration","calibrated_pd_frame")` + `("data","labels")` y reúsa `PerformanceEvaluator`. Para que `validation` valide el challenger, `ml` publica su PD en el **formato `calibrated_pd_frame`** (`("ml","calibrated_pd_frame")`). El cableado de `validation` sobre esa fuente ML (vs la del campeón) es una **decisión de orquestación** del usuario o una **rev. menor aditiva** de SDD-22 (elegir `calibrated_pd_frame` source), **no** una dependencia dura de `ml`; se registra como deuda menor B-VAL. `ml` **posee** la comparación campeón-vs-challenger (SDD-22 no la hace).

## 7. Algoritmos y flujo

**`MLStep.execute(study, rng)` — secuencia canónica.**
1. **Resolver config.** Leer `study.config.ml`; en API programática, exigir `MLConfig`.
2. **Construir `requires`.** Desde `feature_source`; validar CT-1 antes de ejecutar.
3. **Derivar azar.** Obtener la semilla estable del run: `seed = study.seeds.int_seed_for("ml")` (compuesta `[root_seed, hash("ml")]`; invariante de reproducibilidad, no se serializa). `rng` del contrato se usa para el recorte de early stopping.
4. **Leer artefactos.** Copiar defensivamente features (`woe_frame`/`selected_woe_frame`/`data.frame`), `labels`, `splits` y `model.raw_pd_frame`.
5. **Construir X/y por partición.** Seleccionar columnas de features (WoE por default), resolver `partition`/`target` desde `splits`/`labels`; validar clases y filas mínimas.
6. **Recortar validación interna.** De `desarrollo`, separar `X_es`/`y_es` (fracción `validation_fraction`, seeded, estratificado por target) para early stopping; el resto es `X_fit`/`y_fit`. `holdout`/`oot` intactos.
7. **Derivar monotonía.** Según `monotonic.mode`: `from_binning` ⇒ dirección `-1` por variable WoE (WoE↑⇒riesgo↓, porque en SDD-06 event rate↑⇒WoE↓) salvo que el binning marque tendencia invertida; `explicit` ⇒ mapa del usuario; `off` ⇒ sin constraints. Traducir a la codificación del backend en el **orden de columnas** de X.
8. **Construir el backend.** `resolve_backend(cfg.backend)`; import perezoso de la librería; `build(params, seed, n_threads, monotone_constraints)`. Si falta el extra ⇒ `MissingDependencyError` con el extra exacto.
9. **Fitear.** `backend.fit(estimator, X_fit, y_fit, eval_set=(X_es,y_es), early_stopping_rounds=...)`. GBDT registran `best_iteration_`.
10. **Predecir PD.** `predict_proba` sobre `predict_partitions`; materializar `pd_hat` alineado por índice → `pd_frame`.
11. **Calibrar (opcional).** Si `calibrate_challenger`, construir un `raw_pd_frame` con `pd_hat` como `pd_raw` y su logit como `linear_predictor`, y correr `PDCalibrator.from_config(...)` (reúso SDD-10) → `calibrated_pd_frame`.
12. **Comparar.** Construir el frame analítico del campeón (`pd_raw` de `raw_pd_frame`) y del challenger (`pd_hat`), y llamar a `PerformanceEvaluator.evaluate(frame, score_column=..., pd_column=..., target_column=..., partition_column=...)` para **cada uno**; para `psi`, `StabilityEvaluator.evaluate(...)`. Diferenciar `discriminant_metrics`/`stability_metrics` por partición → `MLComparisonRecord`.
13. **Importancias nativas.** `backend.feature_importances(estimator)` (gain/split), top-`k`; **no SHAP** (eso es SDD-14).
14. **Validar invariantes.** PD en rango, finitud, índices consistentes, monotonía aplicada donde se pidió, determinismo declarado.
15. **Construir DTOs.** `MLBackendMetadata`, `MLCardSection` (con `metric_sections` CT-2: curvas de comparación), `MLResult`.
16. **Auditar decisiones.** Backend, semilla, threads, determinismo, constraints, brechas de comparación, calibración.
17. **Publicar artefactos.** Escribir todas las claves `provides` bajo `"ml"`.

**Reúso de evaluators (sin reimplementar métricas).** `ml` **no** recodifica AUC/Gini/KS/PSI: importa perezosamente `PerformanceEvaluator`/`StabilityEvaluator` (SDD-11) y les pasa el frame del challenger y el del campeón con los nombres de columna adecuados. La orientación (score_direction) se maneja como en SDD-11: la PD **es** el risk-score (PD↑⇒riesgo↑). Un test AST impide que aparezcan `roc_auc_score`/KS/PSI reimplementados en `ml` (§11).

**Constraint de monotonía en espacio WoE.** Como binning fuerza WoE monótono con el riesgo (SDD-06 §3: event rate↑ ⇒ WoE↓, un bin con más malos tiene WoE más bajo), la constraint natural por variable es **-1** en el eje WoE: **PD no creciente en WoE**. Esto **es coherente con el principio 4** ("monotonía con el riesgo") y con la interpretación del scorecard. Si el binning reporta una variable con tendencia deliberadamente no monótona, `from_binning` deja esa variable sin constraint (`0`) y lo audita.

**Alternativas descartadas.**
- *AutoML / búsqueda de arquitectura por default:* descartado; v1 es scoring-first, un backend por config. La búsqueda de hiperparámetros es SDD-13.
- *Reemplazar el scorecard:* descartado; ML es challenger, no campeón (regulación).
- *Features crudas por default:* descartado como default; exige política de imputación/valores especiales que hoy resuelve binning. Queda como `data_raw` diferido (FALTA-DATO-ML-1).
- *Usar `holdout` para early stopping:* descartado; contamina la partición de evaluación. Se recorta un set interno de desarrollo.
- *Comparar sobre PD calibrada:* innecesario para discriminación (invariante a monótonas); la comparación usa PD cruda de ambos.
- *SHAP aquí:* descartado; es SDD-14. `ml` expone solo importances nativas.
- *Multihilo por default:* descartado; rompe byte-reproducibilidad de GBDT (§9). Default single-thread.
- *`dict[str, Any]` de hiperparámetros:* descartado; se usan modelos Pydantic tipados por backend (sin `eval`, validables).

**Complejidad / rendimiento.** Dominado por el fit del backend (`O(n_estimators · N · depth)` en GBDT). Single-thread encarece el wall-clock a cambio de determinismo; el modo performance (multihilo) queda disponible pero marcado no byte-reproducible. `ml` registra `n_rows_fit`, `n_features`, `best_iteration` y `fit_seconds` en diagnostics.

## 8. Casos borde y manejo de errores

- **Falta el extra del backend:** `MissingDependencyError("instale nikodym[xgboost]")` (o el extra que corresponda), al construir el backend.
- **Falta artefacto `requires`:** `ArtifactNotFoundError` por CT-1 antes de ejecutar.
- **`feature_source` sin su fuente en el DAG:** validación pre-run CT-1 falla con el contrato incumplido.
- **Desarrollo con una sola clase o filas insuficientes:** `MLDataError` (respetando `require_both_classes`).
- **`validation_fraction` deja `X_es` vacío o de una sola clase:** `MLDataError`; se sugiere bajar la fracción o desactivar early stopping.
- **Índices incompatibles campeón↔challenger:** `MLComparisonError` con el tamaño de la diferencia.
- **Monotonía pedida en backend sin soporte:** `MLMonotonicError` si `on_unsupported="error"`; si `"warn"`, se ignora y se audita.
- **Backend produce PD fuera de `[0,1]` o no finitas:** `MLPredictError`; no se clipa silenciosamente.
- **`deterministic=True` con `n_threads>1`:** `MLConfigError` (o coacción a 1 con warning según D-ML-4).
- **`data_raw` con `NaN` y backend que no los tolera sin política:** `MLConfigError`/`FALTA-DATO-ML-1`.
- **CatBoost/LightGBM/XGBoost con versión no soportada:** `MLBackendError` con la versión detectada; los golden values están pineados a versión (§9, §11).
- **Fit no converge / error interno del backend:** `MLFitError` envolviendo la excepción nativa, con backend y semilla.
- **`filterwarnings=["error"]`:** los warnings de los backends se convierten en diagnostics explícitos o excepciones controladas; no se silencian.

Toda excepción propia desciende de `NikodymError`; mensajes en español e incluyen backend, semilla, partición, feature y valor observado cuando aplique.

## 9. Reproducibilidad y auditoría

- **Componente estocástico.** `ml` **es** el consumidor natural del azar del pipeline (a diferencia de `model`/`scorecard`, que hacen `del rng`). El recorte de early stopping usa `rng`; cada backend se siembra con `seed = SeedManager.int_seed_for("ml")` (entropía compuesta `[root_seed, hash_estable("ml")]`; nunca `hash()` builtin sensible a `PYTHONHASHSEED`).
- **Siembra por backend.** `random_state`/`seed`/`random_seed` según la librería; además, para byte-reproducibilidad de GBDT: XGBoost `nthread=1`; LightGBM `deterministic=True` + `force_row_wise=True` + `num_threads=1`; CatBoost `thread_count=1`; sklearn `n_jobs=1`. `apply_global()` propaga `PYTHONHASHSEED` como en el resto del núcleo.
- **Caveat GBDT multihilo.** ESPECIFICACIONES advierte que los GBDT multihilo **no son 100% deterministas** (ESPECIFICACIONES.md:L77). Por eso `deterministic=True`/`n_threads=1` es el default y condición de los golden values; con `deterministic=False` se marca el resultado como **no byte-reproducible** en `backend_metadata` y en el card, y se audita.
- **Determinismo esperado.** `(data_hash + config_hash + root_seed + backend_version + single_thread) → pd_frame, comparison, importances, card idénticos`. Con multihilo solo se garantiza estabilidad estadística, no byte-a-byte.
- **Versión del backend pineada.** Los golden values dependen de la versión exacta de la librería (algoritmos de histograma/orden pueden cambiar). `backend_version_` se registra y los tests golden fijan el rango de versión (D-ML-10); un cambio de versión que mueva el golden es un evento auditado, no un fallo silencioso.
- **Normalización numérica.** `-0.0 → 0.0`; hashes auxiliares (si se hashean frames) con `pandas.util.hash_pandas_object(...).astype("<u8")`, nunca bytes Parquet ni `hash()` builtin.
- **Núcleo liviano.** `import nikodym.ml` no importa `sklearn`/`xgboost`/`lightgbm`/`catboost`/`pandas`/`numpy`; solo registra `MLConfig` + `MLStep` y puebla el hook. Imports pesados dentro de `execute`/adaptadores. Anotaciones de DataFrame/estimador bajo `TYPE_CHECKING` o strings. `pandera` siempre `import pandera.pandas as pa` dentro de función.
- **Audit trail (`log_decision`).** Registrar como mínimo:
  - `ml_backend`: backend, versión, extra, hiperparámetros efectivos;
  - `ml_seed`: semilla derivada, `n_threads`, flag de determinismo;
  - `ml_feature_source`: fuente, nº de features, columnas;
  - `ml_monotonic`: modo, constraints aplicadas y variables sin soporte;
  - `ml_train`: particiones, fracción de early stopping, `best_iteration`;
  - `ml_comparison`: brechas por partición/métrica y `better`;
  - `ml_calibration`: método y parámetros si `calibrate_challenger`, o ausencia;
  - `ml_determinism`: si el modo no es byte-reproducible (warning).
- **Card / report.** `MLCardSection.metric_sections` debe incluir `"comparison_curves"` (campeón vs challenger por partición), `"feature_importances"` (nativas) y `"determinism"` cuando el modo no sea byte-reproducible. `MLResult.term_structure()` retorna `None` (ml no es multi-período; CT-2).
- **Lineage.** `ml` consume `config_hash` y hashes auxiliares aguas arriba; agrega el hash de su `pd_frame`/comparación, no reemplaza el lineage base.

## 11. Estrategia de tests

Marco transversal en SDD-24. Cobertura objetivo 100% para módulos `ml` cuando se implementen (no es módulo regulatorio de piso, pero la marca exige calidad ejemplar). `filterwarnings=["error"]`, `mypy --strict`, ruff `E,F,I,N,UP,B,SIM,RUF,D` y docstrings públicas en español. Los tests de backend van tras markers `requires_xgboost`/`requires_lightgbm`/`requires_catboost`; sklearn (SVM/RF) corre con el extra base.

- **Golden stump hand-verificable.** Un `random_forest`/GBDT de un solo árbol de profundidad 1 sobre un dataset diminuto con un split conocido produce PD de hoja computables a mano (proporción de malos por hoja); se verifica `pd_hat` exacto dentro de tolerancia.
- **Reproducibilidad byte-a-byte.** Dos corridas con misma semilla, `deterministic=True`, `n_threads=1` y mismos datos ⇒ `pd_frame`/`comparison` idénticos tras normalización, por backend.
- **No determinismo declarado.** Con `deterministic=False`/`n_threads>1`, `backend_metadata.deterministic=False` y el card marca el caveat; no se asevera igualdad byte-a-byte.
- **Monotonía respetada.** Con constraint `-1` en una feature WoE, `pd_hat` es **no creciente** al aumentar esa feature (holding otras); backends sin soporte activan `MLMonotonicError`/warning según config.
- **Comparación correcta.** Las métricas del campeón calculadas por `ml` vía `PerformanceEvaluator` coinciden con las que produce `PerformanceStep` sobre el mismo `raw_pd_frame` (consistencia de reúso).
- **No reimplementación de métricas.** Test AST/grep impide `roc_auc_score`/KS/Gini/PSI recodificados en `ml`; solo pueden aparecer como llamada a los evaluators de SDD-11.
- **Extra faltante.** Seleccionar `backend="xgboost"` sin el extra ⇒ `MissingDependencyError("instale nikodym[xgboost]")`.
- **Import guard.** Subproceso verifica que `import nikodym.ml` no deja sklearn/xgboost/lightgbm/catboost/pandas/numpy en `sys.modules`.
- **Config round-trip/hash.** YAML dump/load preserva backend/hiperparámetros; cambiar `backend`/`hyperparameters`/`feature_source` mueve `config_hash`; el default de `NikodymConfig()` con `ml=None` mueve `GOLDEN_DEFAULT_CONFIG_HASH` respecto al golden previo sin `ml` (test no tautológico, cálculo independiente).
- **Resolución de hiperparámetros.** `hyperparameters` de un backend distinto al seleccionado ⇒ `MLConfigError`; `None` instancia los defaults del backend.
- **Early stopping aislado.** Verifica que `holdout`/`oot` no participan del entrenamiento (snapshot de índices usados en fit ⊆ `desarrollo`).
- **Calibración opcional.** Con `calibrate_challenger=True`, `calibrated_pd_frame` tiene el esquema de SDD-10 y `PDCalibrator` se reúsa (no se reimplementa el offset).
- **No mutación.** Snapshots de `woe_frame`, `raw_pd_frame`, `labels`, `splits` permanecen iguales.
- **Endianness/hash.** Cualquier hash auxiliar usa `.astype("<u8")`; grep impide `hash()` builtin en rutas de identidad.
- **Sin SHAP ni Optuna.** Grep/test documental impide importar `shap`/`optuna` en `ml`; esos nombres solo aparecen como frontera hacia SDD-13/14.

Fixtures: `woe_frame_small.parquet` sintético con particiones y monotonía conocida, `raw_pd_frame_small` del campeón alineado por índice, `labels`/`splits` mínimos, `MLConfig` por backend, `FakePerformanceEvaluator`/`FakeStabilityEvaluator` para aislar el reúso, `InMemoryAuditSink`, y datasets degenerados (una sola clase, índices desalineados, PD fuera de rango, backend no soportado, monotonía en backend sin soporte).

## 12. Decisiones abiertas y riesgos

**R0 (Cami).** Ninguno por ahora. `ml` no fija producto irreversible, release público, PyPI, repo-público, deploy ni plata. Las decisiones D-ML son **defaults metodológicos editables** y se listan para revisión de Cami; **la metodología ML no es R0** (así lo pide el encargo de la tarea).

**D-ML para revisión de Cami.**
- **D-ML-1 — Backend default.** Recomendación: `"xgboost"` (GBDT challenger canónico, gana casi siempre en tabular), **default a confirmar por Cami**. Requiere `nikodym[xgboost]`; si se prefiere que el default corra con el extra base `[ml]`, usar `"random_forest"`. (ESPEC enfatiza CatBoost como infrautilizado; es un candidato de default fuerte si Cami prioriza ese sello.)
- **D-ML-2 — Fuente de features default.** Recomendación: `"binning_woe"` (consistencia de pipeline, deps 06/08 exactas). `"selection_woe"` para apples-to-apples estricto; `"data_raw"` diferido.
- **D-ML-3 — Modo de monotonía default.** Recomendación: `"from_binning"` (WoE↑⇒constraint `-1`, es decir PD↓ al subir WoE, por la convención WoE de SDD-06 §3), coherente con principio 4; `svm`/`random_forest` la ignoran con warning.
- **D-ML-4 — Determinismo default.** Recomendación: `deterministic=True`/`n_threads=1` (byte-reproducibilidad y golden values); modo performance multihilo disponible pero marcado no byte-reproducible. Abierta: ¿`deterministic=True`+`n_threads>1` debe fallar (recomendado) o coaccionar a 1 con warning?
- **D-ML-5 — Calibrar el challenger por default.** Recomendación: `False` (la comparación de discriminación no lo necesita); `True` si se quiere alimentar validación/report con PD calibrada.
- **D-ML-6 — Manejo de desbalance.** Recomendación: `class_weight="none"` por default (no reponderar); `"balanced"` disponible.
- **D-ML-7 — Fracción de early stopping interno.** Recomendación: `0.2` de `desarrollo`, seeded, estratificado; `holdout`/`oot` intactos.
- **D-ML-8 — Comparar sobre PD cruda.** Recomendación: sí; discriminación es invariante a la calibración, así que la comparación usa `pd_raw` vs `pd_hat` sin calibrar.
- **D-ML-9 — Hiperparámetros default por backend.** Recomendación: los conservadores de §5 (árboles poco profundos, LR bajo, regularización > 0); de todos modos son el punto de partida del tuning de SDD-13. **A confirmar por Cami.**
- **D-ML-10 — Estrategia de golden values por versión de backend.** Recomendación: pinear el rango de versión de la librería para los golden; un cambio de versión que mueva el golden es un evento auditado (no fallo silencioso).
- **D-ML-11 — Wrapper target-agnóstico para LGD/EAD.** Recomendación: diseñar la abstracción `Backend`/`MLChallenger` sin acoplarla a clasificación binaria (permitir un `objective`/`task` futuro), pero **implementar solo PD/binario en v1**; LGD/EAD entran cuando F4 los requiera.

**FALTA-DATO explícitos.**
- **FALTA-DATO-ML-1 — Política de imputación para `data_raw`.** Features crudas con missing/special exigen política declarada por variable antes de alimentar SVM/RF; sin ella, `data_raw` no se habilita.
- **FALTA-DATO-ML-2 — Umbral de "challenger materialmente mejor".** La brecha de AUC/Gini que justifica adoptar/monitorear el challenger es decisión de gobierno; no se inventa, entra por config del usuario.
- **FALTA-DATO-ML-3 — Garantías de determinismo cross-versión de cada backend.** Documentadas como caveat; los golden se pinean a versión.

**Riesgos y mitigaciones.**
- **ML confundido con modelo primario.** Mitigación: frontera dura en §1 (challenger, no reemplazo); el campeón sigue siendo el scorecard regulatorio.
- **Comparación no apples-to-apples.** Mitigación: mismas features/particiones/etiquetas, mismo `PerformanceEvaluator`, PD cruda de ambos.
- **No determinismo GBDT tomado por bug.** Mitigación: default single-thread + flag explícito + golden pineados a versión.
- **Reimplementar métricas por inercia.** Mitigación: reúso de evaluators SDD-11 + test AST anti-reimplementación.
- **Fuga de `tuning`/`explain` hacia `ml`.** Mitigación: fronteras a SDD-13/14 en §1 + test anti-import `optuna`/`shap`.
- **Backends pesados en import.** Mitigación: imports perezosos por adaptador + test `sys.modules`.
- **Contaminar `holdout`/`oot` con early stopping.** Mitigación: recorte interno seeded de `desarrollo`; test de aislamiento de particiones.
- **Golden values frágiles cross-versión.** Mitigación: pineo de versión + auditoría de cambios de golden (D-ML-10).
