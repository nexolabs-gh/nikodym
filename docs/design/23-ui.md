## 1. Propósito y responsabilidad

**Qué resuelve (una frase).** `ui` es la **app web premium** que deja construir, ejecutar y explorar un modelo del repo **sin escribir código**: un **SPA React** genera **automáticamente** su formulario desde el schema Pydantic v2 de `NikodymConfig` (sin duplicar validación ni lógica), un **backend FastAPI** invoca la **API pública** (`nikodym.run`) para calcular y **serializa** lo que el motor produjo, y el front lo muestra con charts premium — cumpliendo ESPEC §4.5 ("el config Pydantic ES el experimento; la UI es su editor visual") y §4.6 ("un solo núcleo, dos interfaces").

**Responsabilidad única (qué SÍ hace).**
- **Editor del config declarativo (front).** Genera el formulario desde `NikodymConfig.model_json_schema()` + metadatos `json_schema_extra` (`ui_widget`/`ui_group`/`ui_order`, SDD-05 §5.5); **no** escribe un formulario a mano por sección (DRY, ESPEC §4.6).
- **Validación por reconstrucción, no reimplementación (backend).** La verdad de validación es Pydantic: el backend reconstruye `NikodymConfig.model_validate(dict)` y devuelve el `ValidationError` estructurado; **nunca** reimplementa rangos `ge/le`, `Literal`, uniones, tuplas ni finitud (ESPEC §4.6; SDD-05 §5). El front solo **muestra** los errores.
- **Round-trip YAML↔modelo (reúso de SDD-05).** Carga un YAML existente (con migración, SDD-05 §5.4) para poblar el formulario y exporta el YAML canónico del config editado; **reúsa** `load_config`/`dump_config` de SDD-05.
- **Runner sobre la API pública mínima.** Ejecuta la corrida invocando **exclusivamente** `nikodym.run(config)` (que internamente hace `assemble_run` + `Study(config).run()` + publicación de inventario, CT-4). Cero cálculo propio.
- **Serialización de artefactos a JSON (`serializers.py`).** Solo formatea DTOs/frames; no recalcula.
  La excepción exacta y externa al serializador es `reliability.py`, descrita en el punto siguiente.
- **Visor de resultados y reportes (front).** Renderiza el `ModelCard` consolidado, los `*CardSection`/DTOs de cada dominio y el HTML determinístico de `report` (SDD-26) — **solo formatea** lo ya materializado, con charts premium.
- **Derivación de presentación acotada.** `reliability.py` es la única excepción: deriva bins,
  Brier/ECE y Wilson de forma determinista y read-only para la curva de confiabilidad. No modifica
  el `Study`, no alimenta modelo/`ModelCard`/informe y no redefine outputs contractuales (§6).
- Provee un pequeño **`UiConfig`** de ajustes de la *app* (tema, modo de despliegue, límite de subida, directorio de trabajo, secciones expuestas) **separado** de `NikodymConfig` y **fuera** del `config_hash`.

**Frontera dura de responsabilidad (qué NO hace, y quién lo hace).**
- **CERO recálculo de outputs contractuales, regulatorios o de modelado.** No recalcula binning,
  WoE, PD, ECL, provisiones ni métricas oficiales: todo eso pasa por `nikodym.run`/`Study`. La única
  derivación propia es la vista de confiabilidad de presentación, explícita y trazada (§6).
- **El backend NO importa módulos de dominio** (`binning`/`selection`/`model`/`calibration`/…). Lee el schema (`core.config`) y los artefactos **genéricamente** (los DTOs Pydantic saben `model_dump`-earse a sí mismos; los DataFrames, `to_dict("records")`). Esto evita ciclos y mantiene la frontera de ESPEC §6.1.
- **La UI no añade una sección `ui` a `NikodymConfig` ni contribuye al `config_hash`.** No es un dominio del pipeline: no hay `@register`, no entra a `_DEFAULT_DOMAIN_ORDER`, no mueve `GOLDEN_DEFAULT_CONFIG_HASH`. Es una **herramienta**, no un `Step` (D-UI-3). `UiConfig` queda fuera del hash.
- **No reimplementa validación, migración, serialización ni identidad del config.** Reúsa SDD-05 (schema, round-trip, migración, `config_hash`).
- **No orquesta el pipeline.** El orden de pasos, los prerequisitos CT-1 y el ensamblado sink+inventory los resuelven `core`/`assemble_run`/`nikodym.run` (CT-4); la UI solo dispara `run`.
- **No persiste el audit-trail ni el lineage.** El `Study`/`governance` lo hacen; la UI los presenta.
- **El núcleo NUNCA importa la UI, FastAPI ni React.** `core`/dominios no dependen de `nikodym.ui` (ESPEC §6.1). FastAPI/uvicorn viven tras el extra opcional `[ui]` con import perezoso (ESPEC §4.9).
- **No expone ejecución arbitraria a la red.** Corre **local por defecto**; la demo hosteada NO ejecuta el motor en la nube (corridas **pre-cacheadas**). Exponer ejecución en vivo a Internet es evento sensible = **decisión R0 de Cami** (§12, D-UI-R0).

## 2. Contexto y ubicación en la arquitectura

- **Capa:** Producto (F7/T6). Es la **última capa** y la más externa: se sitúa **por encima** de toda la API pública y no la consume ningún módulo de cómputo. Materializa la "segunda interfaz" de ESPEC §4.6 sobre el mismo núcleo config-driven.
- **Quién la invoca:** el **usuario final no-code** vía navegador. En local, `nikodym-ui`
  (console-script, SDD-25) levanta Uvicorn en `http://127.0.0.1:8000`; la demo hosteada es otro modo
  de despliegue, sin ejecución en vivo.
- **A quién invoca:** el **backend FastAPI** invoca la **API pública** de `nikodym` (`nikodym.run`, `assemble_run`), el schema `NikodymConfig` (SDD-05), el `ModelCardBuilder`/`ModelCard` (SDD-03) y el HTML de `report` (SDD-26). El **front** solo habla con el backend por REST/JSON; nunca importa Python ni ejecuta cómputo.

```text
   navegador (SPA React/Vite + shadcn/ui + charts premium)
        │  REST/JSON
        ▼
   ┌──────────────── nikodym.ui  (backend FastAPI, SDD-23) ───────────────┐
   │  /schema  /validate  /datasets  /run  /results/{id}  /report/{id}     │
   │     │          │          │        │         │             │          │
   │  model_json  model_    synthetic  nikodym  serializers  report.html   │
   │  _schema()   validate   datasets   .run()  (formatea)   (SDD-26)      │
   └─────┼──────────┼──────────┼─────────┼──────────┼─────────────┼────────┘
         ▼          ▼          ▼         ▼          ▼             │ (solo lee artefactos)
   NikodymConfig (SDD-05) ─────► nikodym.run(config) ─► Study.run() ─► artifacts/results/card/report
                                 (assemble_run, CT-4)   (SDD-01)        (SDD-03/26)
```

**Interacción con `Study` y el config declarativo.** La UI **no** es un `Step`: no implementa `execute`, `requires`/`provides`, ni se registra. Su relación con el `Study` es de **cliente de la API**: (1) el front arma un dict JSON-able desde el formulario; (2) el backend lo valida a `NikodymConfig` frozen y llama `nikodym.run(config)`; (3) lee el `Study` resultante (`artifacts`, `results`, `run_context`, `ModelCard`, HTML de report) y lo serializa para el visor. El `config` que produce es **el mismo objeto** que produciría un usuario por código — de ahí la garantía DoD F7 (§9).

**Cableado en `core.study`.** **Ninguno.** La UI no toca `_DOMAIN_MODULES`, `_DEFAULT_DOMAIN_ORDER`, `INFRA_SECTIONS` ni los hooks diferidos `_*_CONFIG_CLS`. Este *no-cableado* es decisión de diseño (D-UI-3). Lo único que toca SDD-25 es el `[project.scripts]` (console-script `nikodym-ui`) y el extra `[ui]`.

**Normativa relevante.** Ninguna directa. La UI **no** contiene parámetros CMF/IFRS 9 ni fórmulas
de riesgo. Su obligación regulatoria es indirecta: **no romper** reproducibilidad ni auditabilidad
— el config que emite debe reconstruir estructuralmente el camino por código (§9), los datos deben
quedar identificados por `data_hash` y todo número mostrado debe tener un artefacto de origen
citable.

## 3. Conceptos y fundamentos

> Esta capa no aporta fórmulas de riesgo; sus "fundamentos" son de **arquitectura de producto** y se anclan en ESPEC.

### 3.1 El config ES el experimento (config-driven)
ESPEC §4.5: el **config Pydantic ES el experimento** y la UI su editor visual. Consecuencia dura: la UI **no tiene estado propio de modelado**; su única salida "de negocio" es un `NikodymConfig` válido. Todo lo que el usuario "hace" se reduce a producir ese objeto y dispararlo. Esto hace demostrable el DoD F7 (§9): dos rutas (código y UI) que producen **el mismo** `NikodymConfig` producen **la misma** corrida.

### 3.2 Un solo núcleo, dos interfaces (generación automática de formulario)
ESPEC §4.6: la UI **genera su formulario desde el mismo schema y no duplica lógica**. El mecanismo es la introspección de `NikodymConfig.model_json_schema()` (Pydantic v2, JSON-Schema Draft 2020-12 con `$defs`/`$ref`/`discriminator`/`anyOf`) más los metadatos de presentación `json_schema_extra` (`ui_widget`/`ui_group`/`ui_order`, contrato de SDD-05). El front consume ese schema y aplica el mapeo tipo→widget (§5). Un sub-config sin metadatos `ui_*` cae al **default por tipo** (defendible); ningún campo queda sin render.

### 3.3 Validación por reconstrucción (una sola fuente de verdad)
La verdad de validación es Pydantic, **en el backend**: `NikodymConfig.model_validate(dict)`; el `ValidationError` se devuelve **estructurado** (lista de `{loc, msg, type}`) y el front lo pinta inline junto al campo. La UI **nunca** reimplementa `ge/le`, `Literal`, uniones ni finitud (doble fuente de verdad = riesgo regulatorio). El front puede dar *hints* de UX (p.ej. cotas del slider tomadas del schema) pero **la validación autoritativa es del backend**.

### 3.4 Round-trip YAML↔modelo (reúso de SDD-05)
Carga: `NikodymConfig.model_validate(yaml.safe_load(text))` tras migración (SDD-05 §5.4). Guardado: `dump_config` de SDD-05 (YAML canónico). Identidad: `config_hash` (SDD-05 §5.5), recomputado en vivo al editar para dar feedback sin ejecutar nada.

### 3.5 Frontera front/back y el modelo de estado
El SPA React mantiene el **config en construcción como un objeto JSON en el estado del cliente**
(store React) y delega **toda** validación e identidad al backend por REST. Como `NikodymConfig` es
**frozen** (SDD-05), no hay mutación en sitio: el dict editable ⇄ modelo frozen re-validado calza
exactamente con el round-trip de SDD-05. El backend es **stateless entre requests** salvo la
persistencia de corridas por `run_id` (`runs.py`).

### 3.6 Testing headless del contrato
La verificación unitaria del **backend** no depende de un navegador:
`fastapi.testclient.TestClient` ejercita los endpoints y `serializers.py` se testea al 100 % como
lógica pura. El DoD de producto sí usa un **candidate wheel fuera del checkout** y Playwright:
instala solamente `wheel[ui]`, arranca `nikodym-ui --no-open`, carga `/` y sus assets, recorre F1 con
un CSV propio y los presets recomendados F1/F3/F4 hasta `done` + HTML, y bloquea toda red que no sea
loopback (§11).

## 4. API pública (contrato)

### 4.1 API pública del motor: extensión aditiva `nikodym.run(config, *, data=None)`
La superficie `run(config)` ya implementada conserva su comportamiento. B2 propone una extensión
aditiva, experimental y keyword-only para entregar un frame preparado sin falsear el contrato de
carga ni crear otra ruta de orquestación (CT-4):

```python
# nikodym/api.py  (junto a assemble_run; api.py YA importa audit/governance/tracking en top-level)
# Path/pandas solo bajo TYPE_CHECKING; las anotaciones son diferidas.
def run(
    config: NikodymConfig,
    *,
    data: str | Path | pd.DataFrame | None = None,
) -> Study:
    """Ejecuta una corrida completa end-to-end.

    Superficie pública única de ejecución (CT-4). Envuelve el ensamblado + el Study + la
    publicación de inventario. Tras superar el preflight devuelve un ``Study`` inspeccionable:
    finalizado (``run_context.status == "done"``) o fallido (``"failed"``) con lineage.

    Los errores del preflight de ``data`` explícita levantan ``NikodymError`` antes de crear
    Study o abrir un sink. Solo los ``NikodymError`` de ``Study.run`` se capturan como Study
    fallido (D-UI-2); el consumidor debe chequear ``study.run_context.status``.
    """
    frame = None
    if data is not None:                              # PRE-FLIGHT: todavía no hay Study/sink
        from nikodym.data import DataConfig, DataLoader  # importa pandas; solo en este branch
        from pydantic import ValidationError as PydanticValidationError
        try:
            data_cfg = (
                config.data
                if isinstance(config.data, DataConfig)
                else DataConfig.model_validate(config.data)
            )
        except PydanticValidationError as exc:
            raise ConfigError(f"Config de datos inválido: {exc}") from exc
        if data_cfg.load.source is not None:
            raise ConfigError("data requiere config.data.load.source=None")
        frame = DataLoader.from_config(data_cfg.load).load(data)  # path o DataFrame; copia propia
        frame = _apply_index_contract(frame, data_cfg.schema_.index_col)

    sink, inventory = assemble_run(config)          # FanOutSink + Null/MLflowInventory (CT-4)
    governance_cfg = _governance_config(config.governance)
    study = Study(config)
    study.set_audit_sink(sink)
    try:
        if frame is not None:
            # Después de set_audit_sink: la inyección queda trazada por ArtifactStore.
            study.artifacts.set("data", "input_frame", frame)
        try:
            study.run()                             # re-levanta tras marcar failed + lineage
        except NikodymError:
            return study                            # solo errores nacidos dentro de Study.run
    finally:
        _close_audit_sink(sink)                     # éxito, fallo capturado o excepción inesperada
    if governance_cfg is not None and governance_cfg.publish_to_inventory:
        entry = _inventory_entry_from(study)        # via ModelCardBuilder(governance_cfg).build(study)
        inventory.register(entry)                   # idempotente por (model_name, config_hash); Null=no-op
    return study
```

**Semántica y decisiones del contrato:**
- **Dos fases de fallo (precisión de D-UI-2).** El preflight de una entrada `data` explícita
  coacciona `config.data` a `DataConfig`, valida la fuente única, carga y resuelve el índice **antes
  de `assemble_run`/`Study`**; un error levanta `ConfigError`/`DataValidationError` y no deja sink
  abierto ni un `Study` ficticio. Superado ese límite, `Study.run()` sigue siendo *fail-loud*
  (marca `status="failed"`, conserva lineage, emite `run_end{failed,error}` y re-levanta), mientras
  `nikodym.run` captura ese `NikodymError` y devuelve el `Study` parcial. El consumidor por código
  DEBE chequear `study.run_context.status`; `_close_audit_sink` corre siempre una vez que el sink fue
  creado. Esto permite al backend distinguir preflight ⇒ 422 de corrida iniciada fallida ⇒ 200.
- **Publicación de inventario.** Solo en éxito y solo si `governance.publish_to_inventory`. Se construye la `ModelCard` con `ModelCardBuilder(config.governance).build(study)` y de ahí una `InventoryEntry` que se registra vía `inventory.register(entry)` (Protocol `ModelInventory`, `@runtime_checkable`, idempotente por `(model_name, config_hash)`; `NullInventory` es no-op). El mapeo exacto `ModelCard → InventoryEntry` se fija contra `governance/inventory.py` al codificar (B23.1) — es detalle de implementación, no de contrato.
- **`run_id`.** Lo genera `Study.run()` (`uuid4().hex`), queda en `study.run_context.run_id`. El backend lo usa como clave de persistencia y en `/api/results/{run_id}`. No es reproducible ni entra al hash (identificador de ejecución).
- **Entrada aditiva.** `data` es keyword-only y acepta `str | Path | DataFrame | None`. `None`
  conserva exactamente la semántica actual. Cualquier valor explícito coacciona perezosamente el
  posible blob `config.data` a `DataConfig` **sin mutar el config original** y exige
  `data_cfg.load.source is None`. Tanto path como DataFrame pasan por
  `DataLoader.from_config(data_cfg.load).load(data)`; el cargador aplica su copia defensiva y sus
  reglas de formato/backend/opciones.
- **Índice e inyección.** Se usa el atributo Python real `data_cfg.schema_.index_col`. Sin
  `index_col`, el frame se conserva. Con él, solo hay dos estados válidos: (a) una columna de ese
  nombre y un `RangeIndex` por defecto, que se valida no-nula/única y se promueve con `set_index`;
  o (b) un índice simple ya nombrado exactamente así, sin columna homónima, no-nulo/único, que se
  conserva. Columna + cualquier índice nombrado/no trivial, `MultiIndex`, nombre distinto o ambas
  representaciones simultáneas son ambigüedad y levantan `DataValidationError`. Tras
  `set_audit_sink`, la API inyecta `("data", "input_frame")`, puerta prioritaria de `DataStep`, para
  que la escritura quede auditada. El pipeline conserva schema, lineage y `data_hash`.
- **Núcleo liviano.** `Path` y pandas viven solo bajo `TYPE_CHECKING`/anotaciones diferidas;
  `DataConfig`/`DataLoader` —y con ellos pandas— se importan dentro de `data is not None`. Acceder a
  `nikodym.run` o invocarlo sin `data` explícita no carga `nikodym.data`.

**Export perezoso en `__init__.py` (PEP 562).** `api.py` importa `audit`/`governance`/`tracking` en top-level; re-exportar `run` de forma *eager* rompería el núcleo liviano. Se usa `__getattr__`:

```python
# nikodym/__init__.py
__all__ = ["__version__", "run", "assemble_run"]
_LAZY = frozenset({"run", "assemble_run"})

def __getattr__(name: str) -> object:          # PEP 562: solo se dispara al ACCEDER el atributo
    if name in _LAZY:
        from nikodym import api                 # importa audit/governance/tracking recién aquí
        return getattr(api, name)
    raise AttributeError(f"module 'nikodym' has no attribute {name!r}")

def __dir__() -> list[str]:
    return sorted({*globals(), *_LAZY})
```

**Test obligatorio (núcleo liviano por capas, §9/§11):** tras `import nikodym`, `sys.modules` **no** contiene `nikodym.api`, `nikodym.audit`, `nikodym.governance`, `nikodym.tracking` ni `fastapi`; al **acceder** `nikodym.run` aparecen `nikodym.api`+audit/governance/tracking; recién al **invocar** `nikodym.run(config)` (→ `study.run()`) se importa perezosamente el stack ML de dominio (`optbinning`/`sklearn`/…). Tres tiers, frontera verificada.

### 4.2 Contrato REST del backend (FastAPI)

> Todos JSON salvo `/report` (HTML). Identificadores en inglés técnico; mensajes en español.

| Método | Ruta | Request | Response (2xx) | Errores |

| POST | `/api/validate` | `{ config }` | `{ valid: bool, config_hash: str\|null, errors: [{loc,msg,type}] }` | **siempre 200** (validar es su función; config inválido ⇒ `valid:false`) |

| POST | `/api/config/from-yaml` | `{ yaml }` | `{ config, config_hash }` — config cargado y **migrado** (`loads_config`, SDD-05 §5.4) | **422** si YAML malformado / migración / validación (mensaje del motor) |

**Manejo de errores (contrato):**
- **`ValidationError` de Pydantic:** en `/validate` ⇒ 200 `{valid:false, errors}` (estructura de `exc.errors()`); en `/run` ⇒ **422** con el mismo detalle (config válido es precondición de correr). *Se resuelve así la aparente contradicción del brief: 422 aplica donde un config válido es requisito, no al chequeo de validez.*
- **Corrida fallida:** `nikodym.run` devuelve el `Study` con `status="failed"`; `/run` responde **200** `{run_id, status:"failed"}` y `/results/{run_id}` sirve el `ModelCard` parcial + lineage + el mensaje del `NikodymError` de dominio. **Nunca** un 500 opaco.
- **`MissingDependencyError` depende de la fase, sin rango ambiguo.** Si nace en el preflight
  explícito/DataLoader —p.ej. falta `[excel]` o `[polars]` para cargar el raw— ocurre antes de
  `Study`, se propaga y `/api/run` responde **422** con el mensaje del motor. Si nace dentro de un
  step de `Study.run()` —p.ej. falta `[scoring]`— hereda `NikodymError`, `nikodym.run` la captura
  después de que `Study` marca `failed`/lineage, y el endpoint responde **200**
  `{run_id,status:"failed"}`. En ninguno de los dos casos se enmascara el mensaje.
- **Modo sin ejecución:** `allow_live_execution=false` deniega con **403 tanto `/api/upload` como
  `/api/run`**. Schema, presets, datasets ya disponibles, validación, YAML, resultados y reportes
  siguen siendo legibles; el flag no se reduce a una indicación visual del frontend.
- **Subida raw y agnóstica (`/api/upload`):** no recibe config, no parsea y no convierte a Parquet.
  Escribe por chunks los bytes originales en un path interno seguro
  `uploaded_<sha256><suffix-admitido>`; el basename del usuario nunca decide el path. La respuesta
  no inventa schema: solo `dataset_id`, nombre de presentación, `size_bytes` y `file_suffix`.
- **Carga al ejecutar (`/api/run`):** la UI reconstruye el config, localiza el raw, crea una copia
  efectiva cambiando **solo** `data.load.source=None` —campo de infraestructura excluido del hash— y
  llama `nikodym.run(effective_config, data=raw_path)`. La capa fina `api.run`, no `nikodym.ui`,
  carga con `DataLoader.from_config(config.data.load)`, honra `file_format`/`backend`/`csv_options`,
  valida/aplica `index_col` e inyecta el frame. Parse/config/índice inválido se traduce a 422 del
  endpoint.
- **Sin normalización intermedia.** Los bytes raw permanecen intactos; no se escribe Parquet ni se
  altera formato/backend/opciones para entregar el frame. `data_hash` identifica el contenido lógico
  que consumió el pipeline (§9).
- El backend **no importa dominios**: resuelve schema y artefactos genéricamente (§6).

**Contrato HTTP de la SPA.** La aplicación se sirve en `/`, consume una API **relativa** bajo
`/api`, carga assets bajo `/assets` y opera same-origin. `1.6.0` no soporta despliegue bajo subpath.
FastAPI registra todas las rutas `/api` **antes** del fallback de la SPA; el fallback solo resuelve
rutas de navegación del cliente y nunca captura un `/api/*` desconocido. Antes de bind, el servidor
parsea `static/index.html`, resuelve **todos** los `src`/`href` locales —JS, CSS, favicon y cualquier
otro recurso— dentro de `static/` y falla si falta, escapa el root o no es un archivo regular.

**Seguridad local obligatoria.** El launcher fija el bind a `127.0.0.1` —no existe `--host`— y
genera `secrets.token_bytes(32)` por lanzamiento. El token de 256 bits no se registra ni se incluye
en la URL: el servidor lo inyecta **en memoria** al responder el index, sin modificar
`static/index.html`; esa respuesta lleva `Cache-Control: no-store` y la SPA envía el token en un
header dedicado. Middleware valida en **todas** las
requests el `Host` exacto `127.0.0.1:<puerto-efectivo>`; `/api/upload` y `/api/run` exigen además
`Origin` exactamente same-origin y token válido. CORS externo permanece deshabilitado.

## 5. Configuración (schema Pydantic) y mapeo tipo→widget

La UI maneja **dos** configs de naturaleza distinta:

**(a) El artefacto editado = `NikodymConfig` (reúso de SDD-05).** Sujeto principal de la UI. La UI **no lo define**: lo lee (`model_json_schema()`), lo puebla en el formulario y lo emite. Schema, validación, round-trip, migración e identidad (`config_hash`) son **propiedad de SDD-05/01**; la UI los **consume**. Que la UI dependa del schema y no lo duplique es el corazón de ESPEC §4.6.

**(b) `UiConfig` (ajustes propios de la app).** Modelo Pydantic pequeño (subclase de
`NikodymBaseConfig`) que parametriza **la herramienta**, no el experimento (§4.3). Vive en
`nikodym/ui/settings.py`, se carga de `ui.yaml`/env, y **jamás** entra a `NikodymConfig` ni al
`config_hash`. Tema, modo y `workdir` no alteran la identidad del experimento; la ubicación de datos
es infraestructura excluida del `config_hash`, mientras el contenido consumido queda trazado por
`data_hash` (D-UI-3).

**Sin sección `ui` en `NikodymConfig` y sin hook diferido.** A diferencia de `validation`/`report`, no se agrega `ui` a `NikodymConfig`, no hay `_UI_CONFIG_CLS`, no hay `field_validator`. La UI es transversal-externa (D-UI-3): mantiene el núcleo liviano y evita un objeto sin correlato de cómputo dentro del hash.

**Mapeo canónico tipo Pydantic → widget (contrato del front; `web/src`).**

| Tipo del campo (schema JSON) | Widget (shadcn/ui) | Afford / cotas |
|---|---|---|
| `Literal[...]` / `Enum` | `Select` | opciones = `enum` del schema |
| `bool` | `Switch` / `Checkbox` | — |
| `int`/`float` con `ge` **y** `le` | `Slider` (+ input numérico) | `min=ge`, `max=le`, `step` por tipo |
| `int`/`float` sin ambas cotas | `Input[type=number]` | `min`/`max` = cota presente |
| `str` | `Input` / `Textarea` | placeholder del `description` |
| `tuple[Literal[...], ...]` | multiselect (grupo de checkboxes) | opciones = `enum` de los items |
| unión discriminada por `type` | `Select`(`type`) + **subform condicional** | resuelve `discriminator` + `$ref` de la variante; render **solo** de la variante elegida |
| sub-modelo anidado (`$ref`) | `Accordion` recursivo | agrupa por `ui_group` |
| `X \| None` (sección opcional, `anyOf` con `null`) | `Checkbox` "activar" | alterna `None` ⇄ default |
| tipo exótico / no mapeado | **editor JSON validado** (Monaco/CodeMirror) | `POST /api/validate` antes de aceptar; marcado visible (§8) |

El `title`/`description` (obligatorios en SDD-05) alimentan etiqueta y tooltip; `ui_widget` en `json_schema_extra` **sobreescribe** el default por tipo cuando existe.

**Ejemplo trabajado (unión discriminada `model`).** La sección `model` es `Annotated[Union[LogitModelConfig, XGBoostModelConfig], Field(discriminator="type")]` (SDD-05). El schema la emite como `oneOf`/`discriminator{propertyName:"type", mapping}`. El front: (1) `Select("type", ["logit","xgboost"])`; (2) si `logit`, renderiza **solo** los campos de `LogitModelConfig` (`stepwise`, `p_value_max` slider, `enforce_sign`…); (3) si `xgboost`, **solo** los de `XGBoostModelConfig`. Cambiar `type` reescribe el subform; `POST /api/validate` resuelve la unión por `type` (falla temprano si el tag no está registrado).

| `calibration` | `reliability` | curva de confiabilidad **derivada** en la capa UI (`nikodym/ui/reliability.py`, B35a) del frame `("calibration","calibrated_pd_frame")`; `None` si el frame falta. Shape: `{"strategy":"quantile","n_bins":10,"by_partition":[{"partition","n","brier","ece","bins":[{"bin","n","mean_predicted_pd","observed_default_rate","ci_low","ci_high","pd_lo","pd_hi"}]}]}`. Binning por deciles de igual frecuencia (`pandas.qcut(pd_calibrated, n_bins, duplicates="drop")`); Wilson 95 % (z=1.96) sobre la tasa observada; `brier = mean((pd_calibrated-target)^2)`, `ece = Σ (n_bin/n_part)·|pred-obs|`; particiones con `n=0` excluidas; orden canónico dev/holdout/oot. El motor V1 no se toca (el cálculo vive en la capa UI, solo pandas/numpy) |
| `performance` | `deciles`, `discriminant` | `("performance","performance_table")` (`DecilePerformanceRecord`) → records; `("performance","discriminant_metrics")` (`DiscriminantMetricRecord`, solo el punto KS: `auc`,`gini`,`ks`,`ks_cutoff_risk_score`,`ks_cutoff_score`,`tpr_at_ks`,`fpr_at_ks`,`source`,`status`) → records |

Fuera de alcance de B27 (no se emiten): `correlation_matrix`/`vif_table`/`stability` de selección; vectores ROC/KS completos (el motor solo tiene el punto KS). La **curva de fiabilidad observado-vs-predicho** SÍ se emite desde B35a como `calibration.reliability` (derivada en la capa UI, módulo `nikodym/ui/reliability.py`; ver la fila correspondiente arriba).

**Invariantes (duras).**
- **Identidad del experimento.** Para el mismo experimento,
  `config_UI == config_código` por igualdad estructural y
  `config_hash(config_UI) == config_hash(config_código)`. Tras ejecutar, además coinciden
  `data_hash` y los resultados canónicos. El `config_hash` no identifica bytes del archivo ni su
  ubicación (DoD F7, §9).
- **La UI nunca escribe bajo namespaces de dominio** en `study.artifacts` ni muta artefactos (lee copias/DTOs frozen).
- **Frontera numérica.** La UI no recalcula outputs contractuales, regulatorios ni de modelado.
  `reliability.py` es la única derivación determinista de presentación (bins/Brier/ECE/Wilson):
  read-only, trazada al frame fuente, no muta el `Study` ni alimenta modelo, `ModelCard` o informe.
  Toda nueva derivación requiere reapertura y revisión del SDD.
- **Nunca `NaN`/`Inf` en el JSON.** El motor ya los evita; el serializador añade un **guard defensivo**: si detecta un no-finito, levanta `UiSerializationError` (falla ruidoso) en vez de emitir tokens `NaN`/`Infinity` que rompen parsers JSON estrictos.
- **`UiConfig` ∉ `config_hash`** y ∉ `NikodymConfig`.
- **Sin `eval`.** Editar el config solo compone estructuras validadas por Pydantic (el mini-DSL `Predicate`/`Rule` de `data` es libre de `eval`, SDD-02).

## 7. Algoritmos y flujo

**Flujo de una sesión (alto nivel).**
1. **Boot (`nikodym-ui`).** `argparse` resuelve bind fijo `127.0.0.1:8000`, `.nikodym_ui/`, apertura
   del navegador y `--no-open`/`--port`/`--workdir` (sin `--host`). Valida el index y todos sus
   `src`/`href` locales —favicon incluido—, genera el token efímero de 256 bits, configura
   Host/Origin/token y recién entonces levanta Uvicorn. El token se inyecta en memoria en el index
   servido; nunca se persiste, loguea ni incorpora a la URL.
2. **Config.** El front pide `GET /api/schema`, arma el formulario (§5), y en cada edición hace `POST /api/validate` → pinta `config_hash` en vivo o errores inline. Round-trip YAML: subir → validar/poblar (con migración SDD-05); descargar → YAML canónico.
3. **Datos.** `GET /api/datasets` lista los sintéticos; `/api/upload` valida únicamente
   tamaño/suffix y persiste bytes raw por chunks, sin parsear ni conocer el config. Con
   `allow_live_execution=false`, la lectura permanece disponible pero upload/run responden 403.
4. **Ejecutar.** `POST /api/run {config, dataset_id}`: el backend (a) reconstruye y valida el config;
   (b) resuelve el raw path; (c) copia el config cambiando solo `source=None`; (d) llama
   `nikodym.run(config, data=raw_path)`, que carga con `DataLoader`, aplica el índice e inyecta el
   frame; (e) persiste el `Study` y responde. No hay Parquet intermedio.
5. **Resultados.** `GET /api/results/{run_id}`: `runs.load(run_id)` → `serialize_study(study, governance=...)` (§6) → JSON. El front lo renderiza con charts premium; secciones ausentes → "no disponible", nunca fabricadas.
6. **Reporte.** `GET /api/report/{run_id}`: sirve el HTML determinístico de `report` (SDD-26) si la corrida lo produjo.

**Resolución de `dataset_id`, `config_hash` y `data_hash`.** El endpoint resuelve `dataset_id` a un
raw path interno, pero la copia efectiva conserva todo el contrato de carga y cambia solo
`data.load.source=None`. Como `source` es infraestructura excluida, su `config_hash` coincide con el
config del experimento. `api.run` carga el path según ese contrato y `data_hash` se calcula sobre el
frame lógico preparado; nombre, ubicación y bytes de un Parquet inexistente no sustituyen esa
identidad.

**Contrato del DoD F7.** Antes de ejecutar, el config producido por la UI y el construido por código
deben ser estructuralmente iguales tras la normalización de infraestructura y compartir
`config_hash`. Después de ejecutar sobre el mismo CSV propio, deben compartir `data_hash` y
resultados canónicos. Esta comparación separa correctamente identidad del experimento, contenido y
ubicación; no usa identidad byte-a-byte del Parquet ni considera equivalente solo el hash del
config.

**Decisiones algorítmicas y alternativas descartadas.**
- *Formulario a mano por sección:* **descartado** (viola ESPEC §4.6; se desincroniza con cada campo nuevo). Se genera desde `model_json_schema()`.
- *Reimplementar rangos/enums/finitud en front o back:* **descartado** (doble fuente de verdad = riesgo regulatorio). Validación = reconstruir el modelo.
- *Backend que llama a `Study`/`assemble_run`/dominios directamente:* **descartado**; llama `nikodym.run` (una sola superficie, CT-4).
- *Backend que importa DTOs de dominio para serializar:* **innecesario**; los DTOs Pydantic se auto-`model_dump`-ean; el serializador es domain-agnostic.
- *Ejecución asíncrona / cola de trabajos en v1:* diferida; síncrona basta para el pipeline F1 sobre datos de ejemplo. Async/cola cuando haya corridas largas concurrentes (v2).
- *Streamlit / Gradio / Dash:* **descartado** — no venden a un gerente C-level (ROADMAP §F7). React premium es producto co-estratégico.

**Complejidad / rendimiento.** Schema y validación son O(nº de campos); despreciable. El costo real es la **corrida** (delegada al motor). La serialización es O(tamaño de artefactos). Sin paralelismo propio.

## 8. Casos borde y manejo de errores

- **Falta el extra `[ui]` (fastapi/uvicorn no instalado):** `UiDependencyError` "instale `nikodym[ui]`" al arrancar; `import nikodym`/`import nikodym.core` **nunca** lo dispara (núcleo liviano).
- **Build estático incompleto:** falta `static/index.html` o cualquier `src`/`href` local (incluido
  favicon) ⇒ el
  launcher falla con rutas accionables **antes de bind**; no levanta un backend REST que parezca una
  UI sana.
- **Host inesperado:** no hay host configurable: el proceso solo enlaza `127.0.0.1`. Un header
  `Host` distinto de `127.0.0.1:<puerto>` se rechaza antes del routing.
- **Origin/token inválido o ausente:** `/api/upload` y `/api/run` rechazan la request; CORS externo
  no está habilitado. El token jamás aparece en logs, archivos o URL.
- **Index con token cacheable:** prohibido; respuesta sin `Cache-Control: no-store`, escritura a
  disco o reutilización tras reinicio falla los tests.
- **`allow_live_execution=false`:** upload y run responden 403; schema, presets, validación,
  resultados y reportes siguen accesibles.
- **Run sobre raw inválido:** formato/opciones incompatibles, parse fallido o índice ausente,
  nulo/duplicado producen 422 de `/api/run`. `/api/upload` ya exitoso permanece agnóstico e
  inmutable.
- **Config editado inválido:** `/validate` ⇒ 200 `{valid:false, errors}` (inline en el front); `/run` ⇒ **422**. El botón Ejecutar queda deshabilitado hasta que valide.
- **YAML con `schema_version` no soportado:** propaga el error de migración de SDD-05 (`MigrationNotFoundError`/`ConfigVersionError`); no migra en silencio.
- **`dataset_id` desconocido:** `UiDatasetError` ⇒ **404**.
- **Tipo de campo no mapeado:** el front cae al **editor JSON validado** para ese subárbol (`POST /api/validate` antes de aceptar), marcado visiblemente; **nunca** se descarta ni se envía sin validar.
- **Corrida que falla a mitad de pipeline:** `nikodym.run` captura el `NikodymError`, devuelve el `Study` con `status="failed"` + lineage; `/run` ⇒ 200 `{status:"failed"}`; `/results` sirve el card parcial + el mensaje del dominio. El backend **no** se cae; **nunca** un 500 opaco.
- **Artefactos ausentes (corrida parcial):** cada sección del visor muestra "no disponible"; jamás se inventan métricas ni se rellenan `NaN`.
- **No-finito colado en un artefacto:** `UiSerializationError` claro (defensivo); el motor ya los evita.
- **Falta un extra al correr:** `MissingDependencyError` de DataLoader/preflight ⇒ **422** antes de
  `Study`; la misma clase nacida dentro de `Study.run` (p.ej. `[scoring]`) ⇒ `Study failed` + lineage
  y **200**. El mensaje "instale `nikodym[<extra>]`" nunca se enmascara.
- **Concurrencia / multiusuario:** v1 asume **un usuario local**. Estado compartido / multiusuario / ejecución expuesta en red = **R0** (D-UI-R0).

Toda excepción propia desciende de `NikodymError`; mensajes en español con la acción, el campo/artefacto afectado y la causa.

## 9. Reproducibilidad y auditoría

- **Componentes estocásticos.** **Ninguno** propio de la UI (salvo el `uuid4` del `run_id`, no reproducible por diseño, fuera del hash, y el dataset sintético que se genera **seeded** = determinista). La reproducibilidad se **delega íntegramente** al config + la API (SDD-01/05).
- **Invariante DoD F7 (garantía central).** Un modelo F1 completo construido 100 % desde la UI es
  idéntico al construido por código cuando ambos producen el mismo `NikodymConfig` por **igualdad
  estructural** y el mismo `config_hash`. Esa precondición no basta para identificar datos: después
  de ejecutar se comparan también `data_hash` y resultados canónicos. `data_hash` representa
  contenido lógico; ni el hash de bytes del Parquet ni su ubicación sustituyen este contrato
  (SDD-01/02/05; ROADMAP §F7).
- **La UI no contribuye al lineage ni al `config_hash`.** No es un `Step`. Lo que **sí** garantiza es que el YAML descargable **reconstruye** la corrida por código (round-trip fiel de SDD-05).
- **Auditoría.** La corrida emite su propio audit-trail/lineage/`ModelCard` (SDD-01/03); la UI **los presenta**, no los genera. `nikodym.run` publica en el inventario (SR 11-7) solo si `governance.publish_to_inventory` y solo en éxito (un modelo fallido no entra al inventario).
- **Determinismo del render.** `serializers.py` **no** altera números: solo
  `model_dump`/`to_dict` sobre DTOs ya materializados. La única excepción, explícita y externa al
  serializador, es `reliability.py`: deriva la vista read-only de bins/Brier/ECE/Wilson bajo la
  frontera acotada de §6, sin mutar `Study` ni alimentar modelo/ModelCard/informe. El origen citable
  (dominio, clave) acompaña cada tabla.

Convenciones de implementación (backend): `mypy --strict`; ruff regla `D`, docstrings en español; identificadores/API en inglés técnico, textos de UI en español; tests bajo `filterwarnings=["error"]`; **prohibido `eval`**; import de FastAPI/uvicorn **perezoso** y solo en la capa server.

## 11. Estrategia de tests

Marco transversal en SDD-24. Casos específicos:

- **Golden DoD F7 (backend).** `POST /api/validate` con el config F1 completo verifica
  `NikodymConfig.model_validate(dict_UI) == cfg_código` y
  `config_hash(UI) == config_hash(código)`. Tras ejecutar sobre el mismo contenido, verifica
  `data_hash` y resultados canónicos; no usa el hash físico del Parquet.
- **Núcleo liviano por capas (§4.1).** `import nikodym` ⇒ `sys.modules` sin `nikodym.api`/`audit`/`governance`/`tracking`/`fastapi`; acceder `nikodym.run` ⇒ aparecen api+audit/governance/tracking; invocar `nikodym.run(config)` ⇒ aparece el stack ML de dominio. Tres tiers.
- **Contrato de `nikodym.run`.** La forma `run(config)` no cambia. Para cualquier `data` explícita
  (path o DataFrame), coacciona perezosamente un `config.data` ya tipado o blob a `DataConfig` sin
  mutarlo y usa `DataLoader.from_config(...).load(data)`, incluida su copia defensiva. Rechaza
  `source != None`, usa `schema_.index_col`, y prueba por separado: promoción desde columna con
  `RangeIndex` por defecto; índice simple correcto ya existente; nulos/duplicados; columna + índice
  nombrado/no trivial; `MultiIndex`; nombre distinto. El preflight inválido levanta
  `ConfigError`/`DataValidationError` antes de `assemble_run`, sin `Study` ni sink. Tras
  `set_audit_sink`, la inyección `("data","input_frame")` queda trazada; éxito, fallo capturado de
  `Study.run` y excepción inesperada cierran siempre el sink, mientras el fallo iniciado conserva
  status/lineage y no publica inventario. Tests de AST/`sys.modules` prueban que la capa
  `nikodym.api` no importa DataConfig/DataLoader/pandas al acceder a `run` ni antes del branch
  explícito; una ejecución normal puede importarlos después por sus propios steps de dominio.
- **Endpoints (TestClient).** `/schema` devuelve el JSON-Schema; `/validate` config válido ⇒ `{valid:true, config_hash}`, inválido ⇒ 200 `{valid:false, errors}`; `/run` inválido ⇒ 422, `dataset_id` malo ⇒ 404, corrida fallida ⇒ 200 `status:"failed"` (no 500); `/results` desconocido ⇒ 404; `/report` sin reporte ⇒ 404.
- **Fase de `MissingDependencyError`.** Dos dobles controlados prueban la misma clase en fronteras
  distintas: levantada por DataLoader antes de crear `Study` ⇒ 422, sin run/sink persistido;
  levantada por un step dentro de `Study.run` ⇒ respuesta 200 `status:"failed"`, lineage disponible
  y sink cerrado. No se acepta 409 ni una clasificación por nombre de extra.
- **Frontera upload/run.** `/api/upload` conserva bytes exactos y devuelve solo metadata física;
  mocks prueban que no llama pandas/DataLoader. `/api/run` pasa el raw path a la API pública y
  produce 422 para parse/config/índice; no crea Parquet ni importa `nikodym.data` desde `nikodym.ui`.
- **Serializador (lógica pura, cobertura 100%).** Con un `Study` fixture finalizado: cada artefacto F1 (§6) se serializa al shape esperado (goldens); DataFrames → `to_dict("records")`; DTOs → `model_dump(mode="json")`; no-finito ⇒ `UiSerializationError`; `study.artifacts` intacto tras serializar (no-mutación).
- **Delegación de validación (AST + comportamiento).** Fuera de `reliability.py`, `nikodym.ui` no
  contiene fórmulas ni reimplementa rangos/enums/finitud; valor inválido produce el
  `ValidationError` del modelo, no una corrección propia.
- **Frontera de `reliability.py`.** Goldens fijan bins/Brier/ECE/Wilson; tests prueban entrada
  read-only, `Study` intacto y ausencia de escrituras/consumo por modelo, `ModelCard` o informe.
- **Backend domain-agnostic (AST).** `nikodym.ui` importa solo `nikodym` (para `run`), `nikodym.core.config` y `nikodym.governance` (para `ModelCardBuilder`); **no** importa `binning`/`model`/`calibration`/…
- **Round-trip.** Subir YAML → poblar → descargar YAML: igualdad semántica (SDD-05); `schema_version` viejo migra; `type` inválido lista tags.
- **Datasets sintéticos.** `list_datasets` estable; `materialize(id)` determinista (mismo parquet byte-lógico entre corridas, seeded); columnas/`role` consistentes con lo que `config.data` espera.
- **Seguridad.** AST sin `eval`/`exec`; subida por chunks limitada por la única fuente
  `upload_max_mb`; *path traversal* bloqueado; `allow_live_execution=false` ⇒ 403 en upload/run;
  bind fijo a loopback; `UiConfig` ∉ `config_hash`. TestClient y Playwright cubren negativos de
  `Host`, `Origin` y token (ausente, incorrecto y reutilizado tras reinicio); verifica además
  `Cache-Control: no-store` y que el index inyectado no se persiste ni queda en cache.
- **Frontend y supply-chain.** Node y pnpm pineados, instalación con lock congelado,
  lint+typecheck+Vitest+build, procedencia Vite por output/hash, reconciliación de cierres
  declarativos full/prod, notices derivados de procedencia, gate anti-fixtures y
  `git diff --exit-code` más status limpio después de regenerar `static/`. Metadata de licencia
  ausente/ambigua falla.
- **Candidate wheel clean-room (gate B2.4).** Fuera del checkout, instalar **solo** el
  candidate `wheel[ui]`; lanzar `nikodym-ui --no-open`; Playwright admite únicamente loopback y
  falla ante cualquier request externa. Verifica `/` y HTTP 200 para **todo** recurso local
  `src`/`href` del index —incluido favicon— y ejecuta hasta
  `status=done` + resultados + HTML: F1 `f1-estandar-consumo` con `consumo_comportamiento`, F3
  `f3-provisiones-consumo` con `provisiones_consumo` y F4 `f4-ifrs9-retail` con
  `ifrs9_retail_latam`. Además repite F1 con un CSV externo propio que contiene `loan_id` no
  nulo/único. No reutiliza dependencias ni imports del checkout.
- **Control negativo de distribución.** En una instalación descartable del mismo candidate se
  elimina `static/index.html` y, en otro caso, el favicon/u otro recurso local obligatorio; el
  launcher debe fallar antes de bind. Este control prueba el fail-closed, no una ruta alternativa
  del backend.
- **Cobertura (D-UI-10).** **100% de la lógica pura del backend** (`serializers`, `datasets`, `settings`, `routes` sin el arranque de uvicorn); el bootstrap de servidor (`server.create_app`, `__main__`) por smoke de `TestClient`, documentando qué líneas quedan fuera y por qué.

Fixtures: `NikodymConfig` del ejemplo F1 (por código), un `Study` finalizado fixture con artefactos F1 + reporte HTML, `UiConfig` mínimo, YAML de config viejo (migración), datasets sintéticos, utilidades de `TestClient`.

## 12. Decisiones abiertas y riesgos

**Riesgos.**
- **Duplicar validación/lógica del schema.** Mitigación: generación desde `model_json_schema()` + validación por reconstrucción en el backend; test AST que veta reimplementaciones e imports de dominio.
- **`nikodym.run` y el fail-loud del código (resuelto, D-UI-2).** Los errores del preflight de
  `data` se levantan antes de crear `Study`/sink. Una vez iniciada la corrida, el fallo vive ruidoso
  en status/audit/lineage, el consumidor DEBE chequear `run_context.status`, `Study.run()` mantiene
  su re-levantamiento de bajo nivel y el sink siempre se cierra. Ver D-UI-2.
- **DoD de identidad incompleto si solo compara config.** Mitigación: igualdad estructural +
  `config_hash` antes de ejecutar y `data_hash` + resultados canónicos después; Playwright prueba la
  interfaz distribuida y TestClient conserva los contratos unitarios.
- **Dos toolchains (Python + Node).** Mitigación: `web/` es fuente y el build versionado vive en
  `src/nikodym/ui/static/`; un job frontend pineado regenera, audita licencias y exige diff/status
  limpio antes de construir el wheel (D-UI-11).
- **Charts premium con datos regulatorios.** Mitigación: el front solo pinta lo serializado; nada se envía a la capa IA (la narrativa la gobierna SDD-26 con payload sanitizado).
- **Exposición de datos/ejecución si se despliega en red.** Mitigación: **local por defecto**; demo **pre-cacheada** (cero ejecución en nube); ejecución en vivo expuesta = **R0** (D-UI-R0).
- **DNS rebinding/CSRF aun en local.** Mitigación conjunta: bind fijo, `Host` exacto, Origin
  same-origin y token efímero fuera de URL/logs; negativos obligatorios en TestClient/Playwright.
- **Cobertura de un servidor ASGI.** Mitigación: separar lógica pura (100%) del bootstrap (smoke `TestClient`), documentando el gap.

## Decisiones B2.0 aprobadas y R0 pendiente

> El contrato local/distribuible quedó aprobado en B2.0. **D-UI-R0 es la única decisión que Cami
> debe tomar explícitamente antes de exponer ejecución en vivo a la red**; no bloquea B2.1–B2.5.

- **D-UI-1 — Stack.** *Recomendación:* **React + Vite + Tailwind + shadcn/ui + charts premium** sobre **backend FastAPI** (ROADMAP §F7). Streamlit/Gradio/Dash descartados (no venden a C-level; la UI es canal comercial).
- **D-UI-2 — Semántica de fallo de `nikodym.run`. ✅ DECIDIDA (DanIA, 2026-07-06; precisada B2.0).**
  Hay dos fases: preflight de `data` explícita falla con `NikodymError` antes de `Study`/sink; solo
  después, un `NikodymError` de `Study.run()` se captura y `run` devuelve el `Study` fallido con
  lineage. El consumidor por código DEBE chequear `run_context.status`; `Study.run()` conserva su
  re-levantamiento fail-loud y `_close_audit_sink` corre siempre tras crear el sink. Opción futura
  (no ahora): `study.raise_if_failed()`. *Alternativa descartada:* fabricar un `Study` para errores
  de carga/precondición o re-levantar el fallo ya trazado de la corrida con el `Study` adjunto.
- **D-UI-3 — La UI NO es un dominio del pipeline.** *Recomendación:* **sin** sección `ui` en `NikodymConfig`, **sin** `@register`, **sin** aporte al `config_hash`; `UiConfig` separado y no hasheado. Herramienta ortogonal.
- **D-UI-4 — Fuente del formulario.** *Recomendación:* generar desde `model_json_schema()` + metadatos `ui_*` (SDD-05); no formularios a mano (DRY).
- **D-UI-5 — Validación = reconstrucción en el backend.** *Recomendación:* `model_validate` en FastAPI; `/validate` siempre 200 (valid true/false), `/run` 422 si inválido; front solo pinta errores.
- **D-UI-6 — Backend sin recálculo contractual.** *Recomendación:* invoca `nikodym.run` y serializa
  genéricamente; `reliability.py` es la única derivación read-only de presentación y toda ampliación
  requiere revisión SDD.
- **D-UI-7 — Ejecución síncrona en v1.** *Recomendación:* síncrona (basta para F1 sobre datos de ejemplo); async/cola diferido a v2.
- **D-UI-8 — Datasets + identidad.** *Aprobada en B2.0; implementación pendiente:* upload agnóstico por chunks; `/run` reconstruye el
  config con solo `source=None` y pasa el raw path a la extensión aditiva de `nikodym.run`. La API
  pública carga según config, aplica índice e inyecta el frame; no hay Parquet intermedio. Ubicación
  no entra al `config_hash`; contenido lógico sí al `data_hash`.
- **D-UI-9 — Empaquetado.** *Aprobada en B2.0; implementación pendiente:* `[ui]` compone `nikodym[scoring,excel,docx]` y añade
  servidor/multipart; `nikodym-ui` es el primer console script; el build versionado viaja en wheel/sdist y
  ambos se construyen sin Node. HTML obligatorio; PDF degrada explícitamente.
- **D-UI-10 — Cobertura.** *Recomendación:* 100% de la lógica pura del backend (`serializers`/`datasets`/`routes`); bootstrap ASGI por smoke `TestClient`; front por su toolchain.
- **D-UI-11 — CI del frontend.** *Aprobada en B2.0; implementación pendiente:* Node y pnpm pineados, lock congelado,
  lint/typecheck/Vitest/build, procedencia autoritativa por output/hash, reconciliación de
  licencias/notices, gate anti-fixtures y diff/status limpio del build versionado. El candidate
  wheel se prueba luego en clean-room; verificar solo que `static/` existe no basta.
- **D-UI-12 — Seguridad local.** *Aprobada en B2.0; implementación pendiente:* bind fijo `127.0.0.1`; token efímero de 256 bits por
  lanzamiento, inyectado en memoria en el index y enviado por header; `Host` exacto en toda request,
  `Origin` same-origin + token en upload/run y CORS externo deshabilitado. El index inyectado usa
  `Cache-Control: no-store` y nunca se persiste. No hay `--host`.
- **D-UI-R0 — Despliegue con ejecución en vivo expuesta a la red = decisión de Cami.** *Pendiente
  de decisión explícita; fuera de la aprobación B2.0:*
  `1.6.0` fija bind a `127.0.0.1` sin opción `--host` y la demo sirve corridas pre-cacheadas. Exponer
  `/api/run` a red, incluso con `allow_live_execution`, se difiere: requiere confirmación de Cami y
  revisión de autenticación, aislamiento, privacidad y límites de recursos.
