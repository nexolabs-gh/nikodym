## 1. Propósito y responsabilidad

**Qué resuelve (una frase).** `ui` es la **interfaz visual Streamlit** que deja construir, ejecutar y explorar un modelo del repo **sin escribir código**: genera **automáticamente** su formulario desde el schema Pydantic v2 de `NikodymConfig` (sin duplicar validación ni lógica), invoca la **API pública** (`nikodym.run` / `assemble_run` + `Study`) para calcular, y muestra los resultados y reportes ya producidos por el pipeline — cumpliendo el principio ESPEC §4.5 ("el config Pydantic ES el experimento; la UI es su editor visual") y §4.6 ("un solo núcleo, dos interfaces").

**Responsabilidad única (qué SÍ hace).**
- **Editor del config declarativo.** Genera el formulario desde `NikodymConfig.model_json_schema()` + introspección de campos + metadatos `json_schema_extra` (`ui_widget`/`ui_group`/`ui_order`, SDD-05 §5.5); no escribe un formulario a mano por sección (DRY, ESPEC §4.6).
- **Validación por reconstrucción, no reimplementación.** La verdad de validación es Pydantic: la UI reconstruye `NikodymConfig.model_validate(dict)` y **muestra inline** el `ValidationError`; nunca reimplementa rangos `ge/le`, `Literal`, uniones ni finitud (ESPEC §4.6; SDD-05 §5).
- **Round-trip YAML↔modelo (reúso de SDD-05).** Carga un YAML existente (con migración, SDD-05 §5.4) para poblar el formulario y exporta el YAML canónico del config editado; **reúsa** el `dump_config`/`load_config` de SDD-05, no lo reimplementa.
- **Runner sobre la API pública.** Ejecuta la corrida invocando **exclusivamente** la API pública (`nikodym.run(config)`, que internamente llama `assemble_run` + `Study.run`, CT-4); muestra progreso y estado.
- **Visor de resultados.** Renderiza el `ModelCard` consolidado (SDD-03), los DTOs/`CardSection` de cada dominio, las `metric_sections` estructuradas (CT-2) y los `OverlayRecord.payload` (CT-2) leídos de `study.artifacts`/`study.results` — **solo formatea** lo ya materializado.
- **Visor de reportes.** Embebe el HTML determinístico de `report` (SDD-26) y ofrece su descarga; nunca recalcula números.
- Provee un pequeño **`UiConfig`** de ajustes de la *app* (tema, límite de subida, directorio de trabajo, secciones expuestas) **separado** de `NikodymConfig` y **fuera** del `config_hash`.

**Frontera dura de responsabilidad (qué NO hace, y quién lo hace).**
- **CERO lógica de dominio (contrato central).** No calcula binning, WoE, PD, ECL, métricas ni provisiones; **todo** cálculo pasa por `api.py`/`Study`. La UI edita el config y muestra resultados; nada más. Un número en pantalla siempre proviene de un artefacto del pipeline.
- **No añade una sección `ui` a `NikodymConfig` ni contribuye al `config_hash`.** A diferencia de `validation`/`report`, la UI **no es un dominio del pipeline**: no hay `@register`, no hay `_DOMAIN_MODULES["ui"]`, no entra a `_DEFAULT_DOMAIN_ORDER`, no mueve `GOLDEN_DEFAULT_CONFIG_HASH`. Es una **herramienta**, no un `Step` (D-UI-2).
- **No reimplementa validación, migración ni serialización del config.** Reúsa SDD-05 (schema, round-trip, migración, `config_hash`).
- **No orquesta el pipeline.** El orden de pasos, los prerequisitos CT-1 y el ensamblado sink+inventory los resuelven `core`/`assemble_run` (CT-4); la UI solo dispara `run`.
- **No persiste el audit-trail ni el lineage.** El `Study`/`governance` lo hacen; la UI los presenta.
- **El núcleo NUNCA importa la UI ni Streamlit.** `core`/dominios no dependen de `nikodym.ui` (ESPEC §6.1: "la UI importa el núcleo, nunca al revés"). Streamlit vive tras el extra opcional `[ui]` (ESPEC §4.9).
- **No despliega ni distribuye la UI como servicio público por su cuenta.** Corre **local por defecto** (`streamlit run`, localhost); exponerla en un servidor compartido es evento sensible = **decisión R0 de Cami** (§12, D-UI-R0).

## 2. Contexto y ubicación en la arquitectura

- **Capa:** Producto (F7/T6). Es la **última capa** del proyecto y la más externa: se sitúa **por encima** de toda la API pública y no la consume ningún módulo de cómputo. Materializa la "segunda interfaz" de ESPEC §4.6 / §5.9 sobre el mismo núcleo config-driven.
- **Quién la invoca:** el **usuario final no-code**, arrancando `streamlit run nikodym/ui/app.py` o el console-script `nikodym-ui` (SDD-25 posee `[project.scripts]`). No la invoca ningún `Study` ni `Step`.
- **A quién invoca:** la **API pública** de `nikodym` (`nikodym.run` / `assemble_run` + `Study`, CT-4), el schema `NikodymConfig` (SDD-05), el `ModelCard` (SDD-03) y el HTML de `report` (SDD-26); `streamlit` (extra `[ui]`, import perezoso); `pandas`/`numpy` para tabular lo que muestra.

```text
                    ┌───────────────── nikodym.ui (SDD-23) ─────────────────┐
                    │  editor de config   runner        visor resultados     │
usuario no-code ──► │  (form desde schema)(api pública) (cards/metrics CT-2) │
                    │        │               │                 ▲             │
                    └────────┼───────────────┼─────────────────┼─────────────┘
                             ▼               ▼                 │  (solo lee artefactos)
                    NikodymConfig ─► nikodym.run(config) ─► Study.run() ─► artifacts/card/report
                    (SDD-05)         (assemble_run, CT-4)   (SDD-01)        (SDD-03/26)
```

**Interacción con `Study` y el config declarativo.** La UI **no** es un `Step`: no implementa `execute`, `requires`/`provides`, ni se registra. Su relación con el `Study` es de **cliente de la API**: (1) ensambla un `NikodymConfig` frozen a partir del formulario; (2) llama `nikodym.run(config)`; (3) lee el `Study` resultante (`artifacts`, `results`, `card`, HTML de report) para el visor. El `config` que produce es **el mismo objeto** que produciría un usuario por código — de ahí la garantía DoD F7 (§9).

**Cableado en `core.study`.** **Ninguno.** La UI no toca `_DOMAIN_MODULES`, `_DOMAIN_CONFIG_CLASSES`, `_DEFAULT_DOMAIN_ORDER`, `INFRA_SECTIONS` ni el hook diferido `_*_CONFIG_CLS`. Este *no-cableado* es una decisión de diseño (D-UI-2): la UI es ortogonal al pipeline. Lo único que puede tocar SDD-25 es el `[project.scripts]` (console-script) y el extra `[ui]`.

**Normativa relevante.** Ninguna directa. La UI **no** contiene parámetros CMF/IFRS 9 ni fórmulas de riesgo (no hay `FALTA-DATO` regulatorio en esta capa, mismo estatus que SDD-26). Su obligación regulatoria es indirecta: **no romper** la reproducibilidad ni la auditabilidad del núcleo — el config que emite debe ser byte-a-byte el del camino por código (§9).

**Paquete físico troceable.**

```text
src/nikodym/ui/

  exceptions.py       # UiError + subclases (config/upload/render/dependency)
  settings.py         # UiConfig (ajustes de la app; separado de NikodymConfig; fuera del config_hash)
  schema_form.py      # introspección model_json_schema() -> especificación de formulario (lógica PURA, testeable)
  widgets.py          # mapeo tipo Pydantic -> widget Streamlit (Literal/Enum, bool, num ge/le, str, unión, anidado, tupla, opcional)
  config_editor.py    # arma/valida el NikodymConfig desde session_state; carga/guarda YAML (reúsa SDD-05)
  runner.py           # invoca la API pública (nikodym.run / assemble_run + Study); progreso; captura el Study
  results_view.py     # visor de resultados: ModelCard + CardSection + metric_sections (CT-2) + overlays
  report_view.py      # visor de reportes: embebe el HTML de SDD-26; descarga
  app.py              # entrypoint Streamlit: navegación, session_state, cablea editor/runner/visores
```

**Troceo implementable un módulo/commit (B23.x).** Fiel al patrón de los SDD hermanos (SDD-22 §2, SDD-26), cinco bloques `config → generación de formulario desde schema → runner → visor → entrypoint`:
- **B23.1 — settings/exceptions/config-editor base.** `UiConfig` (ajustes de la app), `UiError` y subclases, y el *glue* de round-trip de `NikodymConfig` (carga YAML con migración de SDD-05, `model_validate`, `dump_config`, preview de `config_hash`). **No mueve `GOLDEN_DEFAULT_CONFIG_HASH`** (la UI no aporta sección computacional).
- **B23.2 — generación de formulario desde schema.** `schema_form.py` (introspección pura `model_json_schema()` → *field specs*) + `widgets.py` (mapeo tipo→widget; uniones discriminadas, enums, anidados, tuplas, opcionales; honra `ui_widget`/`ui_group`/`ui_order`). Goldens de *spec* de formulario por sub-config.
- **B23.3 — runner.** `runner.py`: invocación de la API pública, ejecución síncrona con `st.status`/progreso, captura del `Study`/artefactos, manejo de fallo de corrida (conserva lineage del run fallido, SDD-01).
- **B23.4 — visores.** `results_view.py` (ModelCard + CardSection + `metric_sections` CT-2 + `OverlayRecord.payload`) y `report_view.py` (embebe HTML de SDD-26 + descarga). Cero cálculo; solo formateo.
- **B23.5 — entrypoint + DoD F7.** `app.py` (navegación, session_state, cableado), console-script `nikodym-ui`, y el **test end-to-end DoD F7** (construir el config del ejemplo F1 vía `streamlit.testing` → `config_hash` idéntico al del código → corrida byte-a-byte idéntica).

Cada bloque cierra con ruff (regla `D`, docstrings en español), `mypy --strict`, tests del módulo tocado y cobertura según D-UI-12 (100% de la **lógica pura**; *shell* Streamlit por smoke de `AppTest`).

## 3. Conceptos y fundamentos

> Esta capa no aporta fórmulas de riesgo de crédito; sus "fundamentos de dominio" son de **arquitectura de producto** y se anclan en los principios de ESPEC.

### 3.1 Config-driven: el config ES el experimento

ESPEC §4.5 fija que el **config Pydantic ES el experimento** y la UI su editor visual. Consecuencia dura: la UI **no tiene estado propio de modelado**; su única salida "de negocio" es un `NikodymConfig` válido. Todo lo que el usuario "hace" en la UI se reduce a producir ese objeto y dispararlo. Esto es lo que hace demostrable el DoD F7 (§9): dos rutas (código y UI) que producen **el mismo** `NikodymConfig` producen **la misma** corrida.

### 3.2 Un solo núcleo, dos interfaces (generación automática de formulario)

ESPEC §4.6 / §6.2: la UI Streamlit **genera su formulario desde el mismo schema Pydantic y no duplica lógica**. El mecanismo es la introspección de `NikodymConfig.model_json_schema()` (Pydantic v2) más el diccionario de campos (`model_fields`) para tipos que el JSON-Schema aplana. El mapeo canónico (SDD-05 §5.5): `Literal`/`Enum`→selectbox, `bool`→checkbox/toggle, numérico con `ge/le`→slider, `str`→text_input, con el **discriminador** `type` controlando el render condicional de las uniones. Los metadatos de presentación (`ui_widget`, `ui_group`, `ui_order`) viajan en `json_schema_extra` (contrato definido en SDD-05; los widgets se materializan aquí).

**Mapeo canónico tipo Pydantic → widget (contrato de `widgets.py`).**

| Tipo del campo (schema) | `FieldSpec.kind` | Widget Streamlit | Afford / cotas |
|---|---|---|---|
| `Literal[...]` / `Enum` | `select` | `st.selectbox` | opciones = valores del `Literal`/`Enum` |
| `bool` | `bool` | `st.checkbox` / `st.toggle` | — |
| `int`/`float` con `ge` y `le` | `slider` | `st.slider` | `min=ge`, `max=le` |
| `int`/`float` sin ambas cotas | `number` | `st.number_input` | `min`/`max` = cota presente; step por tipo |
| `str` | `text` | `st.text_input` | placeholder del `description` |
| `tuple[Literal[...], ...]` | `multiselect` | `st.multiselect` | opciones = valores del `Literal` |
| unión discriminada por `type` | `union` | `st.selectbox`(`type`) + subform | render condicional de la variante |
| sub-modelo anidado | `nested` | `st.expander` recursivo | agrupa por `ui_group` |
| `X \| None` (sección opcional) | `optional` | `st.checkbox` "activar" | alterna `None` ⇄ `default_factory` |
| cualquier otro | `unsupported` | editor JSON validado | `model_validate` antes de aceptar (§8) |

El `title`/`description` obligatorios de cada campo (SDD-05 §5.3) alimentan la etiqueta y el tooltip; `ui_widget` en `json_schema_extra` **sobreescribe** el default por tipo cuando existe.

**Ejemplo trabajado (unión discriminada `model`).** La sección `model` de `NikodymConfig` es `Annotated[Union[LogitModelConfig, XGBoostModelConfig], Field(discriminator="type")]` (SDD-05 §5.2). El formulario:
1. `selectbox("type", ["logit", "xgboost"])` — el discriminador.
2. Si `logit`: `selectbox(stepwise, ["wald","lr","none"])`, `slider(p_value_max, 0.0, 1.0)`, `checkbox(enforce_sign)` — **solo** los campos de `LogitModelConfig`.
3. Si `xgboost`: `slider(max_depth, 1, 12)`, `checkbox(monotonic)` — **solo** los de `XGBoostModelConfig`.

Cambiar el `type` reconstruye el subform sin arrastrar campos de la otra variante; el dict de sesión se reescribe con la variante elegida, y `model_validate` resuelve la unión por `type` (falla temprano si el tag no está registrado, SDD-05 §8).

### 3.3 Round-trip YAML↔modelo (reúso de SDD-05)

La carga es `NikodymConfig.model_validate(yaml.safe_load(text))` (tras migración, SDD-05 §5.4); el guardado es `yaml.safe_dump(cfg.model_dump(mode="json", by_alias=True), sort_keys=False)`; la identidad es el `config_hash` (SDD-05 §5.5). La UI **consume** estas funciones; no las reimplementa. El `config_hash` en vivo (recomputado al editar) da feedback de identidad al usuario sin ejecutar nada.

### 3.4 Modelo de ejecución de Streamlit (rerun + session_state)

Streamlit **reejecuta el script completo** en cada interacción del usuario. Como `NikodymConfig` es **frozen** (SDD-05 §5.3), no se muta en sitio: el **config en construcción vive como `dict` JSON-able en `st.session_state`** y el modelo Pydantic frozen se **reconstruye bajo demanda** (para validar, previsualizar `config_hash`/YAML y correr). Este patrón (dict editable ⇄ modelo frozen re-validado) calza exactamente con el round-trip de SDD-05 y evita el anti-patrón de "un formulario con su propia copia de la validación".

### 3.5 Testing headless de Streamlit

La verificación no depende de un navegador: Streamlit provee `streamlit.testing.v1.AppTest`, que ejecuta el script, permite fijar valores de widgets y leer el estado resultante. El test DoD F7 (§11) usa `AppTest` para conducir el editor hasta el config del ejemplo F1 y comparar su `config_hash` con el construido por código. **`FALTA-DATO-UI-1`** (menor, de ingeniería, no regulatorio): versión mínima de `streamlit` que garantiza `AppTest` estable y su API exacta, a fijar por render de la doc oficial al cablear B23.2/B23.5.

## 5. Configuración (schema Pydantic)

La UI maneja **dos** configs de naturaleza distinta:

**(a) El artefacto editado = `NikodymConfig` (reúso de SDD-05).** Es el sujeto principal de la UI. La UI **no lo define**: lo lee, lo puebla en el formulario y lo emite. Su schema, validación, round-trip, migración e identidad (`config_hash`) son **propiedad de SDD-05/SDD-01**; la UI los **consume** vía `model_json_schema()`, `model_validate`, `model_dump` y las funciones `load_config`/`dump_config`. Que la UI dependa del schema y no lo duplique es el corazón de ESPEC §4.6.

**(b) `UiConfig` (ajustes propios de la app).** Modelo Pydantic pequeño (subclase de `NikodymBaseConfig` por consistencia: `extra="forbid"`, `frozen=True`, `title`/`description`) que parametriza **la herramienta**, no el experimento. Vive en `nikodym/ui/settings.py`, se carga de un `ui.yaml`/variables de entorno **aparte** del config del estudio, y **jamás** entra a `NikodymConfig` ni al `config_hash`. Campos en §4. Regla dura: cambiar el tema, el límite de subida o el workdir **no cambia la identidad de ningún experimento** (por eso está separado; D-UI-2).

**Sin sección `ui` en `NikodymConfig` y sin hook diferido.** A diferencia de `validation`/`report`, no se agrega `ui: "UiConfig | None"` a `NikodymConfig`, no hay `_UI_CONFIG_CLS`, no hay `@field_validator`. La UI es transversal-externa, no una sección del pipeline (D-UI-2). Esto mantiene el núcleo liviano (ESPEC §4.9) y evita que exista un objeto sin correlato de cómputo dentro del hash.

**Metadatos de presentación (contrato de SDD-05 que la UI materializa).** El mapeo `json_schema_extra={"ui_widget":…, "ui_group":…, "ui_order":…}` lo **define** SDD-05 §5.5 en cada sub-config de dominio; SDD-23 lo **honra** en `widgets.py`/`schema_form.py`. Un sub-config sin metadatos `ui_*` cae al mapeo por tipo (default defendible); ninguno queda sin render.

## 6. Contratos de datos (I/O)

**Inputs de la UI.**

| entrada | origen | uso |
|---|---|---|
| schema de `NikodymConfig` | `NikodymConfig.model_json_schema()` + `model_fields` (SDD-05) | generar el formulario |
| YAML de config (opcional) | subida del usuario / archivo local | poblar el formulario (round-trip SDD-05, con migración) |
| dataset | subida (`st.file_uploader`) o ruta local | se cablea a la sección `data` del config; lo valida SDD-02, no la UI |
| `Study` finalizado | retorno de `nikodym.run` | alimentar los visores |
| `ModelCard` | `governance` (SDD-03) vía el `Study` | visor de resultados |
| HTML de report | artefacto de `report` (SDD-26) | visor de reportes |
| `UiConfig` | `ui.yaml`/env | ajustes de la app |

**Outputs de la UI.**

| salida | destino | nota |
|---|---|---|
| `NikodymConfig` frozen | `nikodym.run(...)` | **el** entregable de negocio; idéntico al del código (§9) |
| YAML canónico del config | descarga | `dump_config` de SDD-05; reproduce la corrida por código |
| corrida lanzada | `Study.run` (vía API) | la UI no ejecuta cómputo propio |
| vistas renderizadas | pantalla Streamlit | efímeras; solo formatean artefactos |
| descargas de artefactos | `config.yaml`, `report.html`, `card.json`, exports (SDD-26/08) | ya producidos por el pipeline |

**Invariantes (duras).**
- **Identidad byte-a-byte.** `config_hash(config_construido_por_UI) == config_hash(config_construido_por_código)` para el mismo experimento (DoD F7, §9). De ahí, por la garantía de SDD-01/05, corrida idéntica.
- **La UI nunca escribe bajo namespaces de dominio** en `study.artifacts` ni muta artefactos (lee copias/DTOs frozen).
- **La UI no produce números.** Todo dato mostrado tiene un artefacto de origen citable (dominio, clave).
- **`UiConfig` ∉ `config_hash`** y ∉ `NikodymConfig`.
- **Sin `eval`.** La edición del config no puede inyectar código: el mini-DSL `Predicate`/`Rule` de `data` es libre de `eval` (SDD-02) y la UI solo compone estructuras validadas por Pydantic.

## 7. Algoritmos y flujo

**Ciclo de la app (`main()` — alto nivel, dado el rerun de Streamlit).**
1. **Boot.** Cargar `UiConfig`; si `streamlit` no está → `UiDependencyError("instale nikodym[ui]")`. Inicializar `st.session_state["config"]` (dict) y `st.session_state["study"]` (None) si no existen.
2. **Navegación.** Pestañas *Config · Datos · Ejecutar · Resultados · Reporte* (según `exposed_sections`).
3. **Config (editor).**
   a. `specs = build_form_spec(NikodymConfig)` (recursivo por sección/anidado; puro).
   b. Renderizar cada `FieldSpec` con su widget (§3.2); el valor leído actualiza `session_state["config"]` (dict).
   c. Uniones discriminadas: `selectbox` sobre el `type` → renderiza **solo** los campos de la variante elegida (render condicional).
   d. Secciones opcionales (`X | None`): checkbox "activar sección" alterna entre `None` y el `default_factory`.
   e. En cada rerun: `cfg = form_state_to_config(state)`; si `ValidationError` → mostrar el mensaje **inline** junto al campo (no se reimplementa la regla); si válido → mostrar `config_hash` en vivo + preview YAML.
4. **Carga/guardado.** Subir YAML → `load_yaml_into_state` (con migración SDD-05) → poblar; botón "descargar YAML" → `config_to_yaml`.
5. **Datos.** Subir/seleccionar dataset (dentro de `upload_max_mb`, formato permitido) → cablear su ruta a `config["data"]`; la validación de esquema es de SDD-02 en la corrida, no de la UI.
6. **Ejecutar (runner).** Construir `NikodymConfig` frozen; `run_study(cfg, progress=st.status(...))` → `nikodym.run(cfg)` (assemble_run + Study.run, CT-4). Guardar el `Study` en `session_state`. Ejecución **síncrona** con progreso (D-UI-7).
7. **Resultados (visor).** `render_results(study)`: `ModelCard` (identidad, propósito, `metrics` escalares, `metric_sections` CT-2, `decisions`, `limitations`, lineage) + `CardSection`/DTOs por dominio + `OverlayRecord.payload`. Secciones ausentes → "no disponible", nunca fabricadas.
8. **Reporte (visor).** `render_report(study)`: embeber el HTML de SDD-26 (`st.components.v1.html`) + descarga.

**`build_form_spec` (pseudocódigo, lógica pura).**
```text
build_form_spec(model_cls, prefix=()):
    specs = []
    for name, field in model_cls.model_fields.items():
        path = prefix + (name,)
        anno = field.annotation
        if is_discriminated_union(anno):            # Field(discriminator="type")
            specs.append(FieldSpec(path, kind="union", discriminator="type",
                                   choices=tags_de_las_variantes(anno), ...))
            # las variantes se expanden al elegir el tag (render condicional), no aquí
        elif is_optional_model(anno):               # X | None con X submodelo
            specs.append(FieldSpec(path, kind="optional", ...))
        elif is_submodel(anno):
            specs += build_form_spec(anno, prefix=path)          # recursión
        else:
            specs.append(spec_por_tipo(name, field, path))       # tabla §3.2
    return ordenar_por(ui_order, orden_de_declaracion)(specs)
```
Es **pura** (sin Streamlit): recibe una clase Pydantic y devuelve `tuple[FieldSpec, ...]`; por eso se testea al 100% con goldens (D-UI-12).

**Decisiones algorítmicas y alternativas descartadas.**
- *Escribir un formulario a mano por sección:* **descartado** (viola ESPEC §4.6, duplica el schema, se desincroniza con cada campo nuevo). Se genera desde `model_json_schema()`.
- *Reimplementar rangos/enums/finitud en la UI:* **descartado** (doble fuente de verdad, riesgo regulatorio). La validación es reconstruir el modelo Pydantic.
- *Mantener un `NikodymConfig` mutable en sesión:* imposible (frozen) e indeseable; se usa dict editable ⇄ modelo re-validado (§3.4).
- *Que la UI llame directamente a `Study`/`assemble_run`/dominios:* **descartado**; llama la **API pública** `nikodym.run` (una sola superficie, CT-4), para no acoplarse a internals.
- *Ejecución asíncrona/con cola de trabajos en v1:* diferida; síncrona con `st.status` basta para el pipeline F1 sobre datos de ejemplo (D-UI-7).
- *Framework alternativo (Gradio/Dash/NiceGUI/React propio):* descartado; el MVP Streamlit es **spec** (ESPEC §5.9/§6.2, D-UI-1).

**Complejidad / rendimiento.** La generación de formulario es O(nº de campos del schema) por rerun; despreciable. El costo real es la **corrida** (delegada al pipeline). El visor es O(tamaño de artefactos a tabular). Sin paralelismo propio.

## 8. Casos borde y manejo de errores

- **Falta el extra `[ui]` (streamlit no instalado):** `UiDependencyError` con "instale `nikodym[ui]`" al arrancar; `import nikodym`/`import nikodym.core` **nunca** lo dispara (núcleo liviano).
- **Config editado inválido:** `ValidationError` de Pydantic → envuelto en `UiConfigError` y mostrado **inline**; el botón Ejecutar queda deshabilitado hasta que el config valide.
- **YAML subido con `schema_version` no soportado:** propaga el error de migración de SDD-05 (`MigrationNotFoundError`/`ConfigVersionError`) como mensaje claro; no migra en silencio.
- **Tipo de campo no soportado por el mapeo de widgets:** `FieldSpec.kind="unsupported"` → se ofrece un **editor JSON validado** para ese subárbol (con `model_validate`), y se marca visiblemente; **nunca** se descarta el campo ni se envía sin validar.
- **Subida inválida (tamaño/formato/ruta):** `UiUploadError`; se bloquea *path traversal* (rutas fuera del `workdir`); el archivo se maneja en memoria/local.
- **Corrida que falla a mitad de pipeline:** se captura el `NikodymError` del dominio, se muestra su mensaje + el **lineage del run fallido** (SDD-01 lo conserva); la app **no** se cae; el `Study` fallido queda en sesión para inspección.
- **Artefactos ausentes en el visor (corrida parcial):** cada sección muestra "no disponible"; jamás se inventan métricas ni se rellenan `NaN`.
- **`metric_sections` presente pero no serializable/renderizable:** `UiRenderError` claro; el fallo es del payload del dominio, no de la UI (la UI no lo repara).
- **Rerun de Streamlit / navegación durante una corrida:** el `Study` vive en `session_state`; re-entrada segura. **Un solo usuario local** es el supuesto de v1; multiusuario/estado compartido es R0.
- **Datos sensibles en pantalla:** el render local muestra lo que el dominio publicó; la UI **no** envía nada a la capa IA (eso lo gobierna SDD-26 con payload sanitizado). Exponer la app en red con datos cargados = R0.

Toda excepción propia desciende de `NikodymError`; mensajes en español, con la acción del usuario, el campo/artefacto afectado y la causa.

## 9. Reproducibilidad y auditoría

- **Componentes estocásticos.** **Ninguno.** La UI no introduce azar ni números; la reproducibilidad se **delega íntegramente** al config + la API (SDD-01/05). La UI no recibe ni usa `rng`.
- **Invariante DoD F7 (la garantía central).** Un modelo F1 completo construido 100% desde la UI es **idéntico** al construido por código, porque ambos caminos producen el **mismo `NikodymConfig`** (igualdad estructural y `config_hash`), y `nikodym.run(mismo config) → misma corrida` por SDD-01/05. **Byte-a-byte** se verifica comparando `config_hash` (equivalente barato y exacto) y, en el test golden, los artefactos resultantes (§11). ROADMAP F7 DoD: "un modelo F1 completo construible 100% desde la UI, idéntico al hecho por código".
- **La UI no contribuye al lineage ni al `config_hash`.** No es un `Step`; no aporta `data_hash`/`config_hash`/decisiones. Lo que **sí** garantiza es que el YAML descargable **reconstruye** la corrida por código (round-trip fiel de SDD-05).
- **Auditoría.** La corrida lanzada emite su propio audit-trail/lineage/model card (SDD-01/03); la UI **los presenta**, no los genera. Opcionalmente registra en el `UiConfig`/log local qué YAML se lanzó, pero la evidencia canónica es el lineage del `Study`.
- **Determinismo del render.** El formateo de la UI **no** altera los números: solo aplica formato de presentación sobre DTOs; el origen citable (dominio, clave) acompaña cada tabla.

Convenciones de implementación: `mypy --strict`; ruff regla `D`, docstrings en español; identificadores/API en inglés técnico, textos de UI en español; tests bajo `filterwarnings=["error"]`; **prohibido `eval`**; import de streamlit **perezoso** y solo en la capa de render.

## 11. Estrategia de tests

Marco transversal en SDD-24. Casos específicos:

- **Golden DoD F7 (el test insignia).** Vía `streamlit.testing.v1.AppTest`: conducir el editor hasta el config del **ejemplo F1 completo** (mismos valores que el fixture por código) → `config_hash(UI) == config_hash(código)` y `form_state_to_config(state) == cfg_codigo` (igualdad estructural frozen). Correr ambos → artefactos byte-a-byte idénticos (o `config_hash` idéntico como equivalente exacto por SDD-05).
- **Generación de formulario (goldens de spec).** Para cada sub-config de dominio: `build_form_spec` produce un `FieldSpec` por campo con el `kind` correcto; `ge/le` → `minimum`/`maximum`; `Literal`/`Enum` → `choices`; unión → `kind="union"` + `discriminator="type"`; anidado → recursión; `ui_widget`/`ui_group`/`ui_order` respetados.
- **Delegación de validación (AST + comportamiento).** Test AST: `nikodym.ui` **no** contiene reimplementaciones de rangos/enums/finitud ni fórmulas de dominio (no `roc_auc`, no WoE, etc.). Comportamiento: un valor inválido produce `ValidationError` reconstruyendo el modelo, mostrado inline; la UI no lo "corrige".
- **Round-trip.** Subir un YAML → poblar formulario → descargar YAML: igualdad semántica con el original (SDD-05); `schema_version` viejo migra; `type` inválido lista tags.
- **Runner solo-API.** Test AST/mocks: `run_study` invoca **únicamente** `nikodym.run`/`assemble_run` públicos; no importa dominios ni toca `Study` internals; corrida fallida → `Study` con `status="failed"` + lineage, sin crash.
- **Visor sin cálculo ni mutación.** Con un `Study` fixture finalizado: cada card/`metric_section` presente se renderiza; ausentes → "no disponible"; snapshots profundos de `study.artifacts` intactos tras el visor.
- **Tipo no soportado.** Un campo con tipo exótico → `kind="unsupported"` → editor JSON validado, marcado; nunca se envía sin validar ni se descarta.
- **Casos borde.** Falta `[ui]` → `UiDependencyError`; subida sobre `upload_max_mb`/formato inválido/ruta fuera de `workdir` → `UiUploadError`.
- **Import liviano.** `import nikodym` / `import nikodym.core` **no** importa streamlit; `import nikodym.ui.schema_form` no exige streamlit; invocar render sin `[ui]` → `UiDependencyError`.
- **Seguridad.** AST sin `eval`/`exec`; *path traversal* bloqueado; `UiConfig` ∉ `config_hash`.
- **Tooling y cobertura (D-UI-12).** `filterwarnings=["error"]`, `mypy --strict`, ruff regla `D`; **cobertura 100% de la lógica pura** (`schema_form`, `widgets` mapping, `config_editor`, shaping de los visores); el *shell* Streamlit (`app.py`, callbacks de render) por **smoke de `AppTest`**, documentando qué líneas de render quedan fuera del 100% y por qué.

Fixtures: `NikodymConfig` del ejemplo F1 (por código), un `Study` finalizado fixture con cards/`metric_sections`/report HTML, `UiConfig` mínimo, YAML de config viejo (para migración), y utilidades de `AppTest`.

## 12. Decisiones abiertas y riesgos

**Riesgos.**
- **Duplicar validación/lógica del schema.** Mitigación: generación desde `model_json_schema()` + validación por reconstrucción; test AST que veta reimplementaciones (ESPEC §4.6).
- **DoD byte-a-byte difícil de probar por una UI.** Mitigación: reducir la prueba a **igualdad de `config_hash`** (exacta y barata) más un golden de artefactos; `AppTest` conduce la app headless.
- **Modelo rerun/session_state de Streamlit propenso a bugs de estado.** Mitigación: dict editable ⇄ modelo frozen re-validado (§3.4); estado mínimo; tests de `AppTest`.
- **UX de corridas largas (bloqueo síncrono).** Mitigación: `st.status`/progreso en v1; async/cola diferido (D-UI-7).
- **Exposición de datos/seguridad si se despliega en red.** Mitigación: **local por defecto**; despliegue en servidor = **R0** (D-UI-R0), con auth/aislamiento fuera de v1.
- **Cobertura 100% de un app Streamlit.** Mitigación: separar lógica pura (100%) del shell (smoke `AppTest`), documentando el gap (D-UI-12).
- **Brechas del mapeo de widgets para tipos exóticos.** Mitigación: fallback a editor JSON validado + marca visible; nunca envío sin validar (§8).

**`FALTA-DATO` (de ingeniería, no regulatorio; a resolver por render de doc oficial al cablear).**
- **FALTA-DATO-UI-1:** versión mínima de `streamlit` con `streamlit.testing.v1.AppTest` estable y su API exacta (fijar el pin y el patrón de test en B23.2/B23.5).

## Decisiones para revisión de Cami

> Ninguna bloquea el diseño; se proponen defaults defendibles. Solo las revisa Cami al final. **D-UI-R0 es la única que Cami debe decidir explícitamente antes de exponer la UI.**

- **D-UI-1 — Framework.** *Recomendación:* **Streamlit**, es el MVP mandado por la spec (ESPEC §5.9/§6.2). Gradio/Dash/NiceGUI/React propio descartados por spec y por acoplar más superficie.
- **D-UI-2 — La UI NO es un dominio del pipeline.** *Recomendación:* **sin** sección `ui` en `NikodymConfig`, **sin** `@register`, **sin** aporte al `config_hash`; `UiConfig` (ajustes de la app) queda separado y no hasheado. La UI es una herramienta ortogonal al pipeline. (Diferencia dura con `validation`/`report`.)
- **D-UI-3 — Fuente del formulario.** *Recomendación:* generar desde `NikodymConfig.model_json_schema()` + `model_fields` + metadatos `ui_*` (SDD-05 §5.5); no escribir formularios a mano (DRY, ESPEC §4.6).
- **D-UI-4 — Validación = reconstrucción del modelo.** *Recomendación:* validar por `NikodymConfig.model_validate(dict)` y mostrar `ValidationError` inline; **nunca** reimplementar `ge/le`/`Literal`/finitud en la UI (una sola fuente de verdad).
- **D-UI-5 — Estado de sesión.** *Recomendación:* config en construcción como **dict JSON-able** en `st.session_state`; `NikodymConfig` frozen reconstruido bajo demanda (compatible con el rerun de Streamlit y con `frozen=True`).
- **D-UI-6 — Mapeo de tipos Pydantic complejos a widgets.** *Recomendación:* `Literal`/`Enum`→selectbox, `bool`→checkbox/toggle, num `ge/le`→slider, `str`→text_input, `tuple[Literal,…]`→multiselect, unión discriminada→selectbox sobre `type` + subform condicional, anidado→expander recursivo, `X|None`→checkbox de activación; tipo exótico→editor JSON validado (fallback marcado, §8).
- **D-UI-7 — Ejecución larga.** *Recomendación:* **síncrona con `st.status`/progreso** en v1 (basta para F1 sobre datos de ejemplo); ejecución en hilo/cola de trabajos diferida a una iteración posterior.
- **D-UI-8 — Runner solo sobre la API pública.** *Recomendación:* la UI llama **exclusivamente** `nikodym.run(config)` (assemble_run + Study, CT-4); cero cálculo propio. DoD F7 verificado por `config_hash` idéntico + golden de artefactos.
- **D-UI-9 — Subida y seguridad de datos.** *Recomendación:* subida **local**, limitada por `upload_max_mb`, en memoria/`workdir`, con *path traversal* bloqueado; el mini-DSL de `data` es libre de `eval` (SDD-02), así que editar el config no inyecta código. La validación de esquema del dataset es de SDD-02 en la corrida, no de la UI.
- **D-UI-10 — Empaquetado.** *Recomendación:* extra **`[ui]`** (streamlit) + console-script **`nikodym-ui`** (SDD-25 posee `[project.scripts]`); import perezoso; `core` nunca importa streamlit. Confirmar el pin de streamlit al cablear (FALTA-DATO-UI-1).
- **D-UI-11 — Visor de reportes.** *Recomendación:* embeber el HTML determinístico de SDD-26 (`st.components.v1.html`) + descarga; nunca recomputar números.
- **D-UI-12 — Cobertura.** *Recomendación:* **100% de la lógica pura** (`schema_form`, mapeo de `widgets`, `config_editor`, shaping de visores); el shell Streamlit por smoke de `AppTest`, documentando el gap. Tratar la lógica pura como código sensible (garantiza el DoD F7).
- **D-UI-R0 — Despliegue/distribución pública de la UI = decisión de Cami.** *Recomendación:* **local por defecto** (`streamlit run`, localhost). Exponer la UI en un servidor compartido / nube pública / multiusuario es **evento sensible R0**: expone superficie de subida y ejecución de config a la red, aloja datos regulatorios en un servidor y no trae auth/aislamiento en v1. Requiere config explícito (`UiConfig.allow_server_deploy`), confirmación de Cami y revisión de privacidad/seguridad antes de habilitarse.
